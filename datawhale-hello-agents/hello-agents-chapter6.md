- Datawhale Hello-Agents: https://github.com/datawhalechina/hello-agents

## 1. 本章主题和学习目标

第六章主题是框架开发实践。

第四章是从底层手写 ReAct、Plan-and-Solve、Reflection 等 Agent 工作流；第五章是体验 Coze、Dify、FastGPT、n8n 等低代码平台如何把底层能力产品化；第六章则进入代码框架视角，学习如何借助主流 Agent 框架更高效、更规范地构建可靠的智能体应用。

本章核心问题：

> 当 Agent 从 demo 走向复杂应用时，为什么需要框架来管理角色、状态、工具、流程、日志、部署和协作？

本章按原文学习了四类代表性框架：

- `AutoGen`: 以对话驱动多 Agent 协作。
- `AgentScope`: 以消息驱动和工程化能力支撑多 Agent 服务系统。
- `CAMEL`: 以角色扮演和引导性提示实现双智能体自主协作。
- `LangGraph`: 以 State、Node、Edge 把 Agent 流程建模成可执行状态图。

关键理解：

> 框架不是为了炫技术，而是把不同类型的复杂性工程化。选框架时不能先问哪个更强，而要先问：这个 Agent 产品最主要的复杂性在哪里？

## 2. 我的学习轨迹总览

这一章的学习不是平铺直叙地记框架名字，而是在不断区分几个容易混的概念。

一开始，我能理解“手写 Agent 不适合大项目”，但理解还偏向“不想重复写代码”。后来纠正为：框架的价值不只是少写代码，而是复用 Agent Loop、解耦模型/工具/记忆、标准化状态管理、提供可观测性，并让复杂系统可维护、可扩展。

我对 AutoGen 的卡点是：为什么要分 `AssistantAgent` 和 `UserProxyAgent`？后来理解为：这是把推理权和执行权分开。`AssistantAgent` 负责想和说，`UserProxyAgent` 代表用户或外部环境发起任务、执行代码/工具、验证结果。

我对 AgentScope 的卡点更深：一开始觉得它和 AutoGen 都是多 Agent 对话，没有本质区别。后来通过“Agent 服务化”理解了它：AgentScope 不是更会聊天的 AutoGen，而是更像多 Agent 服务系统的运行底座，关心消息路由、并发、留痕、恢复、监控和部署。

我最想先学 LangGraph，所以中途主动要求先跳到 LangGraph。虽然这打破了原文顺序，但这次跳转帮助我抓住了第六章最重要的一条产品经理主线：复杂 Agent 产品不是只靠对话推进，还需要显式流程、共享状态、条件分支、失败重试和人工审核。

最后，我把四个框架压缩成一句口诀：

> 开会 AutoGen，底座 AgentScope，双人 CAMEL，流程 LangGraph。

## 3. 6.1 从手动实现到框架开发

第六章开头强调，从一次性脚本走向成熟框架，是软件工程里的重要跃迁。

第四章手写的 Agent 范式适合教学理解，但如果要构建多个、不同类型、逻辑复杂的智能体应用，就会遇到瓶颈：

- 每次都要重复写 Agent Loop；
- 状态、记忆、工具调用要反复处理；
- 日志、调试、异常处理容易混乱；
- 多 Agent 协作时，角色、消息、流程很难靠临时脚本维护。

框架的本质，是提供一套经过验证的开发规范，把智能体共有的重复性工作抽象出来，让开发者更专注业务逻辑。

### 我的关键纠偏

我一开始把“产品级 Agent”概括为框架级别的问题。后来纠正为：产品级 Agent 更具体地需要系统化管理接口、生命周期、状态、调试、扩展边界。

我也一度没有分清状态管理和可观测性。

最终区分为：

- 状态管理回答：Agent 当时知道什么、处在哪一步。
- 可观测性回答：Agent 是怎么一步步跑到这个结果的。

例如：

- 用户已经上传合同，Agent 却又问“请上传合同”，更像状态管理问题，因为它丢了当前流程节点。
- 搜索工具返回错误结果，模型照着用了，更需要可观测性查工具调用参数、返回值、哪一步开始跑偏。

### 最终理解

框架开发不是“换一个更高级的库”，而是把 Agent 从可运行 demo 推向可靠系统。

框架封装的共性工作包括：

- Agent Loop；
- 模型层；
- 工具层；
- 记忆层；
- 状态管理；
- 回调、日志和 trace；
- 多 Agent 协作机制；
- 终止条件和安全阀。

从 AI 产品经理视角看，demo 证明的是能不能做；框架解决的是能不能稳定、可维护、可扩展地做。

## 4. 6.2 AutoGen：对话驱动协作

AutoGen 的核心思想是以对话驱动协作。

它把复杂任务拆成多个可对话的智能体，让它们像一个团队一样自动对话、接力、迭代。原文的软件开发团队案例里，角色包括：

- `ProductManager`: 把模糊需求转成开发计划。
- `Engineer`: 根据计划写代码。
- `CodeReviewer`: 审查代码质量、可读性和健壮性。
- `UserProxy`: 代表用户发起任务，并执行、验证最终代码。

### 我的卡点：UserProxy 和 ProductManager 的区别

我一开始疑惑：UserProxy 也负责提出开发需求，那它和 ProductManager 有什么不同？

后来理解为：

- `UserProxy` 是外部世界的入口，负责把用户原始任务带进系统，也负责执行、验证、反馈。
- `ProductManager` 是团队里的产品角色，负责把原始任务分析成需求、模块、技术建议、验收标准。

前面的 Agent 可能在对话里产出计划、代码和审查意见，但这些产物仍然是文本。`UserProxy` 的价值是把对话产物接到现实执行环境里，例如运行代码、调用工具、验证结果，并通过 `TERMINATE` 结束流程。

### AutoGen 的关键机制

AutoGen 通过 `System Message` 定义每个角色的职责、边界、输出格式和交接方式。

在软件开发团队案例中，`RoundRobinGroupChat` 用固定顺序推进：

```text
ProductManager -> Engineer -> CodeReviewer -> UserProxy
```

其中：

- `participants` 决定参与者和发言顺序；
- `termination_condition` 决定什么时候结束，例如出现 `TERMINATE`；
- `max_turns` 是安全阀，避免多 Agent 无限讨论。

### 我的最终理解

AutoGen 不是让一个大 Agent 全包，而是让开发者先设计一个“可对话团队”：

- 谁负责思考？
- 谁负责执行？
- 谁反馈结果？
- 谁接下一轮？
- 什么时候终止？
- 人类或外部环境在哪里介入？

它适合角色分工明确、需要团队讨论和接力的任务。

它的边界是：

- LLM 对话有不确定性；
- 固定轮询适合流程清楚的任务，但动态流程会显得僵硬；
- 出问题时面对的是长对话历史，对话式调试困难。

## 5. 6.3 AgentScope：多 Agent 服务系统的运行底座

AgentScope 的定位是工程化优先的多智能体平台。

如果 AutoGen 像自动开会的 Agent 团队，AgentScope 更像多 Agent 服务系统的运行底座。

它关心的不是角色怎么聊天，而是：

- 消息怎么传；
- 谁能收到什么消息；
- 消息怎么记录；
- 多个 Agent 能不能并发处理；
- Agent 能不能部署在不同进程或机器上；
- 系统出错后能不能恢复；
- 后台能不能监控运行过程。

### 我的卡点：什么叫 Agent 服务化

我一开始觉得 AgentScope 和 AutoGen 很像，都是多 Agent 和消息。后来通过“服务化”理解了区别。

Agent 服务化不是运行一次脚本，而是把 Agent 变成后台能力：

```text
用户提交任务
-> 系统生成任务/消息
-> 分发给 Agent
-> 处理中间过程被记录
-> 出错可以重试
-> 结果返回用户
-> 管理后台能看到状态
```

服务化 Agent 应该具备：

- 长期在线；
- 任务接入与分发；
- 过程记录与状态管理；
- 故障恢复与扩展；
- 可观测、可运营、可监控。

### 三国狼人杀案例里的理解

原文用“三国狼人杀”展示 AgentScope，因为这个场景比简单问答复杂得多：

- 多个玩家 Agent 同时存在；
- 有白天公开讨论，也有夜晚狼人私聊；
- 有投票、死亡、阶段切换和胜负判断；
- 每个 Agent 既有游戏身份，又有三国人物性格；
- 消息可见范围会随阶段和身份变化。

这个案例让我理解了 `MsgHub` 的价值：它不只是标准化聊天格式，更重要的是做消息路由和通信范围控制。

例如：

- 白天公开讨论发给所有玩家；
- 狼人夜晚私聊只发给狼人；
- 预言家查验结果只发给预言家；
- 玩家死亡后权限可能变化。

如果没有 `MsgHub`，开发者要自己处理谁能看见什么、消息不能泄漏给谁、私聊结束后怎么回到公开频道、每条消息怎么保存等一堆工程逻辑。

### 最终理解

AgentScope 的核心不是让 Agent 更会聊天，而是把多 Agent 应用做成可运行、可监控、可恢复、可部署的工程系统。

它适合大规模、长期运行、高并发、需要留痕和恢复的生产级多 Agent 应用。

它的边界是：

- 对开发者要求更高，需要理解消息驱动、异步编程、分布式通信；
- 简单场景可能过度工程化；
- 生态和社区仍在发展。

## 6. 6.4 CAMEL：双角色扮演协作

CAMEL 的核心是角色扮演和引导性提示。

它最初关注的问题是：

> 能不能在人类少干预的情况下，让两个 AI 通过角色扮演自主协作解决复杂任务？

CAMEL 中常见的两个角色是：

- `AI User`: 懂业务、提需求、下指令、推动任务。
- `AI Assistant`: 执行指令、提供方案、生成内容或代码。

这两个角色不是普通用户和普通助手，而是被模拟出来的两个互补专家。

### 图里的理解：Task Specifier 和 Inception Prompting

CAMEL 图里，人类先给一个粗略想法，例如开发股票交易机器人。

`Task Specifier` 把粗略想法细化成更具体的任务，例如：

> 开发一个带情绪分析工具的交易机器人，能监控社交媒体上关于某只股票的正负面评论，并基于情绪分析结果执行交易。

然后通过引导性提示给两个 AI 注入行动纲领：

- 自己的角色是什么；
- 对方的角色是什么；
- 共同目标是什么；
- 每轮怎么沟通；
- 一次只能提出一个清晰步骤；
- 完成时用特定标志，例如 `<SOLUTION>` 或 `<CAMEL_TASK_DONE>`。

### AI 科普电子书案例

原文案例是让心理学家和作家合作写一本关于“拖延症心理学”的科普电子书。

角色分工是：

- `心理学家`: 提供心理学理论、实证研究、科学解释。
- `作家`: 设计结构、照顾读者体验、把复杂概念转成通俗表达。

这类任务适合 CAMEL，因为它需要两个互补专家反复协作：

- 一方保证专业准确性；
- 一方保证可读性和表达效果。

协作过程自然分成：

- 框架搭建与目标对齐；
- 核心内容生成与知识转译；
- 迭代优化与质量保证；
- 总结与升华。

### 我的关键纠偏

我觉得 CAMEL 和 AutoGen 很像，都是对话协作。后来区分为：

- AutoGen 偏搭团队和编排协作，可以有多个角色、发言顺序、执行者、终止条件。
- CAMEL 偏角色扮演范式，重点是两个互补角色在强提示约束下自主推进任务。

AutoGen 也能做类似的专业/非专业协作，但它的默认抽象是“多角色团队对话”。CAMEL 的默认抽象是“双专家角色扮演 + 引导性提示”。

### 最终理解

CAMEL 是轻架构、重提示。

它适合电子书、方案共创、知识转译、创造性生成等任务。

它的边界是：

- 高度依赖 prompt 质量；
- 出问题时难判断是角色定义、任务描述还是沟通协议的问题；
- 不适合严格流程控制、大规模并发、复杂决策树。

## 7. 6.5 LangGraph：State + Node + Edge 的可执行流程图

LangGraph 是我本章最想重点理解的框架，所以学习时曾主动要求先跳到这一节。

LangGraph 和 AutoGen、CAMEL 的差异是：它不是把任务建模成对话，而是把 Agent 执行流程建模为状态机和有向图。

核心构成：

- `State`: 工作流共享状态，记录当前任务、对话历史、搜索结果、失败次数、风险等级、最终答案等。
- `Node`: 处理步骤，接收 State，执行动作，返回更新后的 State。
- `Edge`: 节点之间的路线，决定做完一个节点后去哪里。
- `Conditional Edge`: 条件边，根据当前 State 动态决定下一步。

### 我的卡点：State、Node、Edge 怎么分

我一开始把“提取条款”理解成 State。后来纠正为：

- 如果它是被记录的数据，多半是 State；
- 如果它是要执行的动作，多半是 Node；
- 如果它是下一步去哪的规则，多半是 Edge 或 Conditional Edge。

合同审核例子：

```text
State:
合同文本、已提取条款、风险等级、人工审核结果、最终报告

Node:
提取条款、判断风险、人工复核、生成报告

Conditional Edge:
风险高 -> 人工复核
风险低 -> 生成报告
```

这个纠偏让我真正理解了 LangGraph 的工作方式：

> Node 负责处理，Edge 负责跳转，State 负责被所有节点读写。

### 三步问答助手案例

原文用三步问答助手展示 LangGraph：

```text
Understand -> Search -> Answer
```

`SearchState` 包含：

- `messages`: 对话历史；
- `user_query`: LLM 理解后的用户需求总结；
- `search_query`: 优化后的搜索关键词；
- `search_results`: 搜索结果；
- `final_answer`: 最终答案；
- `step`: 当前步骤。

我曾觉得这个案例没有很体现 LangGraph。后来确认：原文这个案例主要展示 State、Node、Edge 的基础结构，还没有真正写 `conditional edge`。

搜索失败后的 fallback 是在 `answer` 节点内部用 `if state["step"] == "search_failed"` 判断，而不是图上的条件边。

更能体现 LangGraph 的写法应该是：

```text
Search
-> Conditional Edge
   -> success: Answer With Results
   -> failed: Fallback Answer
```

这个判断很重要：普通流程图也能画“理解、搜索、回答”，但 LangGraph 真正的强项是把分支、回跳、循环、人工审核变成可执行图结构。

### 最终理解

LangGraph 的优势是：

- 流程显式；
- 状态清楚；
- 分支可审计；
- 支持循环和反思；
- 容易插入人工审核；
- 适合需要可靠性和可控性的生产级流程。

它的代价是：

- 前期代码更多；
- 需要定义 State、Node、Edge；
- 对简单问答显得重；
- 开放式协作不如 AutoGen/CAMEL 自然；
- 调试时仍要看节点逻辑、State 更新和条件边判断。

我最终形成的理解是：

> LangGraph 牺牲前期简便性，换来流程可控、状态清楚、分支可审计、循环可实现。

## 8. 四个框架的选型地图

| 框架 | 我最后记住的定位 | 复杂性主要在哪里 | 适合场景 | 主要边界 |
| --- | --- | --- | --- | --- |
| AutoGen | 自动开会的 Agent 团队 | 多角色对话协作 | 软件团队模拟、角色接力、多人讨论 | 对话不确定，调试长对话困难 |
| AgentScope | 多 Agent 服务系统的运行底座 | 消息、并发、留痕、部署、恢复 | 长期运行、高并发、企业级多 Agent 系统 | 工程复杂度高，简单场景可能过重 |
| CAMEL | 双专家角色扮演协作 | 角色设定和引导性提示 | 电子书、方案共创、知识转译 | 强依赖 prompt，不适合严格流程 |
| LangGraph | 可执行流程图 | State、Node、Edge、条件分支 | 合同审核、客服质检、失败重试、人工审核 | 前期代码重，开放式协作较弱 |

最终口诀：

> 开会 AutoGen，底座 AgentScope，双人 CAMEL，流程 LangGraph。

## 9. 对 AI 产品经理的启发

学第六章后，我对“会用框架”和“理解框架边界”的区别更清楚了。

会用框架，是知道它能帮我做什么。

理解边界，是知道引入它会牺牲什么、遮住什么、带来什么新的调试成本。

例如：

- LangGraph 能让流程可控，但要付出更多前期设计和代码成本。
- AgentScope 能支撑服务化运行，但简单 demo 用它可能过度工程化。
- AutoGen 能自然模拟团队协作，但对话跑偏时调试成本高。
- CAMEL 能让双专家共创自然发生，但高度依赖初始提示质量。

作为 AI 产品经理，选框架时应该先问：

1. 这个产品主要复杂在协作、流程、服务运行，还是共创？
2. 需要强可控，还是需要开放式协作？
3. 需要长期运行和并发，还是只是验证原型？
4. 失败后要怎么排查：看状态、看消息、看对话，还是看条件边？
5. 人类在哪些节点必须介入？

## 10. 我的最终理解

第六章让我把前几章串起来了：

- 第四章让我知道 Agent 内部怎么工作；
- 第五章让我看到底层能力如何被平台产品化；
- 第六章让我开始理解 Agent 应用如何被框架工程化。

我最后的判断是：

> 低代码平台不是替代技术理解，框架也不是替代产品判断。平台和框架都在封装复杂性，但 AI 产品经理必须知道复杂性被封装到了哪里，黑盒在哪里，出了问题该从哪里查，什么时候该工程化。

如果以后设计一个合同审核 Agent 产品，我会优先这样拆：

- 用 LangGraph 设计主流程：上传合同、提取条款、风险判断、人工复核、生成报告、失败重试。
- 如果系统要长期运行、并发处理很多合同、每条消息和状态都要留痕，再考虑 AgentScope 做服务化运行底座。
- 如果某个节点需要多角色讨论，例如法务、财务、业务一起评估条款，可以局部使用 AutoGen 思路。
- 如果某个节点需要两个专家共创，例如法律专家和通俗写作助手共同生成用户可读解释，可以借鉴 CAMEL。

这一章最终形成的核心能力不是“背出四个框架”，而是：

> 面对一个 Agent 产品，能判断它的主要复杂性是什么，并据此选择合适的框架抽象和工程边界。
