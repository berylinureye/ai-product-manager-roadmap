## 2. 第三章主题和学习目标

第三章主题是大语言模型基础。它不是直接讲 Agent 产品形态，而是补现代 LLM Agent 的底层能力来源：语言模型为什么能理解上下文、生成文本，并作为现代 Agent 的认知核心。

本章核心线索：

```text
语言模型
-> N-gram 与统计语言模型
-> 神经网络语言模型与词向量
-> RNN/LSTM
-> Transformer
-> Decoder-Only
-> 提示工程、分词、模型调用、模型选择
-> 缩放法则与模型局限
```

本章学习目标：

- 理解语言模型的基本任务：估计词序列或下一个 token 的概率。
- 理解从 `N-gram -> 神经网络语言模型/RNN -> Transformer` 的演进逻辑。
- 理解提示词、分词、模型选择分别在与 LLM 交互时解决什么问题。
- 理解大模型能力提升背后的代价和局限，例如幻觉、成本、可控性和上下文限制。

关键理解：

> LLM 的强大能力不是从“会聊天”开始的，而是从“在上下文中预测下一个 token”这个基础任务逐步发展出来的。

## 3. 语言模型的基础任务

语言模型最基础的任务，不是回答问题，也不是像数据库一样查标准答案，而是：

> 估计一个词序列出现的概率，或在已有上下文下预测下一个 token 的概率。

例子：

```text
我今天去学校学习
学校今天我去学习
```

第一句话通常比第二句话更自然，所以语言模型应该给第一句话更高概率。

生成文本时，模型本质上是在不断做：

```text
给定前面的上下文，下一个最可能出现的 token 是什么？
```

重要区分：

| 方式 | 含义 |
| --- | --- |
| 数据库查询 | 找到已有的标准答案或结构化记录 |
| 语言模型生成 | 基于上下文一步步预测 token，生成看起来合理的回答 |

所以 LLM 的回答有概率性和随机性。它可以生成流畅内容，但不保证一定是事实正确的标准答案。

## 4. 统计语言模型与链式法则

统计语言模型会把一句话的概率拆成一串条件概率相乘。

例如：

```text
datawhale agent learns
```

可以拆成：

```text
P(datawhale agent learns)
= P(datawhale)
  * P(agent | datawhale)
  * P(learns | datawhale, agent)
```

其中 `|` 表示 given，也就是“在......条件下”。

例如：

```text
P(learns | datawhale, agent)
```

表示：

> 在已经看到 `datawhale agent` 的条件下，下一个词是 `learns` 的概率。

链式法则的问题是：如果句子很长，最后一个词可能要依赖前面所有词。

```text
P(第100个词 | 前面99个词)
```

真实语料里，“前面 99 个词完全一样”的情况可能根本没有出现过，因此很难可靠统计。

## 5. 马尔可夫假设与 N-gram

为了解决完整历史太长、无法可靠统计的问题，N-gram 引入了马尔可夫假设：

> 当前词不依赖完整历史，只依赖前面有限几个词。

例如原来想计算：

```text
P(learns | datawhale, agent)
```

如果用 Bigram，也就是 `N=2`，只看前面 1 个词：

```text
P(learns | agent)
```

如果用 Trigram，也就是 `N=3`，看前面 2 个词：

```text
P(learns | datawhale, agent)
```

N-gram 的本质：

> 它不是让模型真正理解语言，而是用局部词组的统计规律近似语言概率。

例子：

```text
我 喜欢 吃 苹果
```

Bigram 会统计：

```text
我 -> 喜欢
喜欢 -> 吃
吃 -> 苹果
```

它看到的是相邻词之间的搭配频率，而不是“我喜欢吃苹果”这个完整语义。

## 6. Bigram 概率的计数估计

Bigram 概率可以通过语料库中的计数估计：

```text
P(w_i | w_{i-1}) = Count(w_{i-1}, w_i) / Count(w_{i-1})
```

读成人话就是：

> 某个词后面接另一个词的概率 = 这两个词连续按顺序出现的次数 / 前一个词出现的总次数。

示例语料：

```text
datawhale agent learns
datawhale agent works
```

估算：

```text
P(datawhale agent learns)
= P(datawhale) * P(agent | datawhale) * P(learns | agent)
```

逐个计算：

```text
P(datawhale) = 2 / 6 = 0.333
P(agent | datawhale) = Count(datawhale agent) / Count(datawhale) = 2 / 2 = 1
P(learns | agent) = Count(agent learns) / Count(agent) = 1 / 2 = 0.5
```

所以：

```text
P(datawhale agent learns) = 0.333 * 1 * 0.5 = 0.167
```

今天纠正的重点：

- `Count(datawhale agent)` 不是“同时出现”，而是“连续、按顺序出现”。
- `datawhale agent` 和 `agent datawhale` 不是一回事。
- `P(learns | agent)` 的分母是 `agent` 出现的总次数，不是 `agent learns` 出现的次数。

## 7. N-gram 的两个核心缺陷

### 7.1 数据稀疏性

如果某个词组没有在语料里出现过，N-gram 会把它的概率估成 0。

例如语料只有：

```text
datawhale agent learns
datawhale agent works
```

那么：

```text
P(sleeps | agent) = Count(agent sleeps) / Count(agent) = 0 / 2 = 0
```

但现实中 `agent sleeps` 不一定不可能，只是这个小语料里没见过。

关键区分：

```text
没见过 != 不可能
```

### 7.2 泛化能力差

N-gram 把词当成孤立符号，不理解词和词之间的语义相似关系。

例如它见过：

```text
agent learns
```

但没见过：

```text
robot learns
```

它不会自动意识到 `agent` 和 `robot` 在某些语境里相似，因此不能把 `agent learns` 的经验迁移到 `robot learns`。

关键理解：

> 更大的语料库可以缓解数据稀疏，但不能从根本上解决 N-gram 的泛化问题。因为 N-gram 的表示方式仍然是词面计数，不是语义表示。

## 8. 神经网络语言模型与词嵌入

为了解决 N-gram 把词当孤立符号的问题，神经网络语言模型引入了一个重要想法：

> 用连续向量表示词。

这个向量叫词嵌入，也叫词向量。

直觉上，每个词不再只是一个离散 ID，而是在高维语义空间里的一个点。

例如：

```text
agent 和 robot 距离比较近
agent 和 apple 距离比较远
```

因为 `agent` 和 `robot` 在很多上下文里更相似，而 `apple` 和它们关系更弱。

神经网络语言模型大致做两件事：

1. 把 token 映射成向量，也就是词嵌入。
2. 根据前面 token 的向量，预测下一个 token 的概率分布。

与 N-gram 的区别：

| 模型 | 主要方式 | 局限或优势 |
| --- | --- | --- |
| N-gram | 统计词面搭配 | 难以泛化到没见过但语义相似的表达 |
| 神经网络语言模型 | 学习连续语义表示 | 更容易利用词之间的相似关系做泛化 |

今天纠正的理解：

> 词向量不是显式地先查同义词，再复制经验；而是让语义相似的词在向量空间中表示接近，神经网络因此更容易把相似模式迁移过去。

## 9. 余弦相似度与词向量关系

词被表示成向量后，就可以用数学方法衡量词与词之间的关系。

余弦相似度的直觉：

```text
两个向量方向越接近，语义越相似。
两个向量方向越不相关，语义关系越弱。
```

大致可以理解为：

```text
1   ：方向几乎一样，强相关
0   ：方向接近垂直，关系弱
-1  ：方向相反，负相关
```

例如：

```text
similarity(agent, robot) > similarity(agent, apple)
```

因为 `agent` 和 `robot` 的语义关系可能比 `agent` 和 `apple` 更接近。

经典例子：

```text
vector("King") - vector("Man") + vector("Woman") ≈ vector("Queen")
```

这个例子说明：

> 词向量空间不只表示单词本身，还可能捕捉一些抽象关系，例如性别、身份、类别等语义方向。

注意：

> 这不代表模型像人一样真正理解王室制度，而是说明词向量空间中可以学到某些可计算的语义关系。

## 10. RNN/LSTM：按顺序传递上下文

词向量解决了“词是孤立符号”的问题，但语言还有顺序问题。

N-gram 只看固定窗口。RNN 的想法是：

> 按顺序读文本，并维护一个隐藏状态，把前文信息逐步压缩和传递下去。

隐藏状态可以理解为：

> 模型读到当前为止的上下文摘要。

例如：

```text
我 昨天 去 图书馆 借 了 一本 ...
```

RNN 会按顺序更新状态：

```text
读到“我” -> 状态更新
读到“昨天” -> 状态继续更新
读到“图书馆” -> 状态继续更新
读到“借” -> 状态继续更新
```

到了预测下一个词时，隐藏状态理论上包含前文信息，所以更可能预测出：

```text
书
```

LSTM 是 RNN 的改进版，主要想缓解：

> 普通 RNN 在长句子里容易忘掉很早之前的信息。

今天纠正的理解：

> RNN 不是每次重新读取所有前文，而是按顺序读，并把前文信息压缩进隐藏状态里逐步传递。

## 11. 为什么还需要 Transformer

RNN/LSTM 虽然比 N-gram 更能处理上下文，但仍然有两个明显问题：

| 问题 | 含义 |
| --- | --- |
| 长距离依赖难 | 早期信息要经过很多步传到后面，中途容易衰减或被遗忘 |
| 并行效率低 | 必须按顺序处理 token，很难同时处理整句话 |

Transformer 的核心变化是：

> 不再主要依赖顺序传递隐藏状态，而是用注意力机制，让 token 在上下文窗口内直接关注其他相关 token。

例子：

```text
这本书虽然封面普通，但内容非常精彩，我很喜欢它。
```

当模型处理“它”时，应该能直接关注前面的“这本书”，而不是完全依赖 RNN 一步步传递过来的隐藏状态。

关键理解：

> Transformer 更适合解决长距离依赖和并行计算效率问题，但它不是无限全局理解。它仍然受上下文窗口、训练数据和模型能力限制。

## 12. Transformer 的 Encoder-Decoder 总体结构

经典 Transformer 来自论文《Attention Is All You Need》，总体结构是 Encoder-Decoder：

```text
输入句子 -> Encoder 编码 -> Decoder 解码 -> 输出句子
```

以机器翻译为例：

```text
输入：I love learning agents.
输出：我喜欢学习智能体。
```

Encoder 负责：

> 处理输入句子，生成带上下文关系的表示。

Decoder 负责：

> 根据 Encoder 提供的信息和已有输出，一个 token 一个 token 生成目标句子。

直觉理解：

```text
Encoder：负责理解输入
Decoder：负责生成输出
```

但这里的“理解”不是人类意识意义上的理解，而是：

> 把输入转成一组可计算的上下文向量表示。

## 13. 输入嵌入与位置编码

Transformer 的输入首先要变成向量。

### 13.1 输入嵌入

输入嵌入做的是：

```text
token -> embedding vector
```

模型不能直接处理文字本身，要先把 token 映射成数字向量。

### 13.2 位置编码

Transformer 不像 RNN 那样天然按顺序读 token。它可以并行处理多个 token，所以如果只给 token 向量，模型不知道谁在前、谁在后。

因此需要位置编码告诉模型：

```text
这个 token 在第 1 个位置
这个 token 在第 2 个位置
这个 token 在第 3 个位置
```

通常 Transformer 的真正输入是：

```text
输入嵌入 + 位置编码
```

例子：

```text
我 喜欢 你
你 喜欢 我
```

这两句话包含一样的词，但顺序不同，含义不同。没有位置编码，模型很难区分它们。

关键理解：

> 输入嵌入告诉模型“这个 token 是什么”，位置编码告诉模型“这个 token 在哪里”。

## 14. Self-Attention 自注意力机制

自注意力机制让模型在处理某个 token 时，可以查看同一句子里的其他 token，并判断哪些 token 对当前 token 更重要。

例子：

```text
这本书虽然封面普通，但内容非常精彩，我很喜欢它。
```

当模型处理“它”时，应该重点关注“这本书”相关 token，而不是只关注离它最近的词。

Self-Attention 里的 `self` 指的是：

> 同一个输入序列内部的 token 互相看彼此。

不是说某个 token 只有 Query，另一个 token 只有 Key 或 Value。更准确地说：

> 每个 token 都会生成自己的 Q、K、V。当前 token 用自己的 Q 去匹配上下文中其他 token 的 K，再根据权重汇总它们的 V。

Self-Attention 相比 RNN 的优势：

- 可以在上下文窗口内直接建立远距离 token 关系；
- 不需要所有信息都一步步通过隐藏状态传递；
- 更适合并行计算。

## 15. Q/K/V 的直觉

Q/K/V 是自注意力机制中的三个核心向量。

| 向量 | 全称 | 直觉理解 |
| --- | --- | --- |
| Q | Query | 我现在想找什么信息？ |
| K | Key | 我这里有什么特征，能不能被别人匹配上？ |
| V | Value | 如果别人关注我，我能提供什么内容？ |

每个 token 都会生成自己的 Q、K、V。

从当前 token 的角度看：

```text
当前 token 的 Q -> 匹配其他 token 的 K -> 得到注意力分数
```

然后：

```text
注意力分数 -> 归一化成注意力权重 -> 用权重加权汇总各 token 的 V
```

用“它”和“这本书”的例子：

```text
“它”的 Query：我在找我指代的对象
“这 / 本 / 书”这些 token 的 Key：我可能是一个可被指代的对象
匹配度高 -> 注意力权重高
“这 / 本 / 书”这些 token 的 Value：提供关于“书”的语义信息
```

今天重点纠正的误区：

- Token 不一定等于一个完整单词，也不是一句话。
- 中文里的“这本书”在真实模型里可能被切成 `这 / 本 / 书` 等多个 token。
- 不是“有的 token 只有 Q，有的 token 只有 K，有的 token 只有 V”。
- 每个 token 都同时有 Q、K、V，只是在一次注意力计算里用途不同。

## 16. 注意力分数、注意力权重与 Value

Q 和 K 匹配后，先得到的是：

> 原始相关性分数，也叫匹配分数。

它还不是注意力比例。

流程是：

```text
Q/K 原始匹配分数
-> softmax/归一化
-> 注意力权重
-> 权重乘以各 token 的 V
-> 加权汇总成当前 token 的新上下文表示
```

注意力权重通常加起来是 1，可以理解成当前 token 对上下文中各 token 分配的注意力比例。

例子：

```text
这: 0.10
本: 0.15
书: 0.45
我: 0.05
它: 0.10
其他: 0.15
```

如果“书”的权重是 `0.45`，“我”的权重是 `0.05`，那么“书”的 V 对“它”的新表示影响更大。

关键理解：

> 权重乘以 V，不是在判断 V 好不好，而是在决定从这个 token 的 V 里拿多少信息。

加权汇总后的结果也不是最终答案，而是：

> 当前 token 更新后的上下文表示。

这个表示后续还会经过更多 Transformer 层、前馈网络、归一化等处理，最后才影响输出 token 的概率。

## 17. Multi-Head Attention 多头注意力

单头注意力的问题不是“看不到全句”，而是：

> 它只有一套 Q/K/V 投影和一套注意力分布，看的角度比较单一。

多头注意力的思想是：

```text
把一次注意力计算分成 h 组，分别计算，再合并。
```

这里的 `h` 表示 head 的数量。

例如：

```text
d_model = 512
h = 8
每个 head 处理的维度约为 512 / 8 = 64
```

`d_model=512` 的意思是：

> 每个 token 在模型内部用一个长度为 512 的向量表示。

它不是 binary，也不是每一维都能被人类明确解释成“颜色 / 主语 / 情绪”。更准确地说，它是模型训练出的隐藏表示维度，混合编码了语义、语法、位置、上下文等特征。

多头注意力的直觉：

```text
同一批 token
-> head 1 用一个表示子空间去看
-> head 2 用另一个表示子空间去看
-> head 3 再从另一个角度去看
-> ...
-> 拼接所有 head 的输出
-> 线性变换整合成统一表示
```

教学上常说：

```text
一个头看指代关系
一个头看语法关系
一个头看位置关系
一个头看语义关系
```

但这不是工程师手写规则。真实训练中，每个 head 学到什么关注模式，是由训练数据和优化目标学出来的。

关键理解：

> 单头不是看得少，而是看的角度少。多头不是看更多 token，而是用多个表示子空间看同一批 token。

## 18. Add & Norm：残差连接与层归一化

Transformer 里常见的 `Add & Norm` 包含两个部分：

```text
Add = 残差连接
Norm = Layer Normalization 层归一化
```

残差连接的做法是：

```text
输出 = 原输入 + 模块处理后的结果
```

例如：

```text
x -> Multi-Head Attention -> attention_output
先做：x + attention_output
```

直觉上，模块不是每一层都从零重建全部信息，而是在原表示上补充和修改信息。

残差连接的好处：

- 原始信息不容易在深层网络中丢失；
- 梯度更容易往前传；
- 训练更稳定。

Layer Norm 的作用是：

> 把每一层的数值分布调得更稳定，避免数值过大或过小，让后续训练更平稳。

一句话总结：

```text
Add：保留原信息
Norm：稳定数值状态
```

`Add & Norm` 不是语言理解的核心机制，而是让深层 Transformer 能稳定训练的重要工程结构。

## 19. Feed Forward 前馈网络

Transformer 层里除了 Attention，还有 Feed Forward Network，简称 FFN。

Attention 和 FFN 的分工：

| 模块 | 作用 |
| --- | --- |
| Multi-Head Attention | 让 token 和 token 之间进行上下文信息交互 |
| Feed Forward Network | 对每个 token 自己的表示做非线性加工 |

FFN 是对每个位置分别处理：

```text
第 1 个 token 的表示 -> FFN -> 更新第 1 个 token
第 2 个 token 的表示 -> FFN -> 更新第 2 个 token
第 3 个 token 的表示 -> FFN -> 更新第 3 个 token
```

它不在这一步直接比较 token 和 token 的关系。不同 token 之间的信息交流主要发生在 Attention 里。

为什么需要 FFN？

> Attention 负责混合上下文信息，但模型还需要非线性变换能力，对每个 token 的表示进行提炼、组合和过滤。

一句话总结：

> Attention 负责横向交流，FFN 负责纵向加工。

## 20. Encoder 层的整体流程

Encoder 核心层可以理解成：

```text
输入 x
-> Multi-Head Self-Attention
-> Add & Norm
-> Feed Forward Network
-> Add & Norm
-> 输出
```

用人话解释：

```text
1. Attention：我和上下文里谁有关？
2. Add & Norm：保留原信息，稳定一下。
3. FFN：我自己这份表示还需要怎么加工？
4. Add & Norm：再次保留原信息，稳定一下。
```

真实 Transformer 会堆叠多层 Encoder。堆叠多层不是重复劳动，而是让 token 表示逐层变得更丰富：

```text
浅层：可能更偏局部搭配、词面关系
深层：可能更偏抽象句法、语义和任务相关模式
```

这些层学到什么，也不是人工规定的，而是在训练中形成的。

## 21. Decoder 层与 Encoder 的不同

Decoder 也有 Attention、Add & Norm、FFN，但它比 Encoder 多一个关键限制：

> Decoder 生成文本时不能偷看未来 token。

因为生成是一个 token 一个 token 往后写：

```text
我 -> 喜欢 -> 学习 -> 智能体
```

当模型正在预测“学习”这个位置或下一个 token 时，不应该提前看到右侧未来 token。否则训练时会作弊，和真实生成条件不一致。

所以 Decoder 里的 Self-Attention 通常是：

> Masked Self-Attention，带遮罩的自注意力。

遮罩的作用：

```text
当前位置只能看自己和前面的 token，不能看后面的 token。
```

例如：

```text
我 喜欢 学习 智能体
```

处理“学习”位置时：

```text
可以看：我、喜欢、学习
不能看：智能体
```

在经典 Encoder-Decoder 架构里，Decoder 还会有 Cross-Attention：

```text
Decoder 当前生成状态 -> 关注 Encoder 的输出
```

机器翻译里就是：

```text
Encoder 读英文输入
Decoder 生成中文时，不断关注英文输入中对应的信息
```

关键理解：

> Encoder 负责双向理解输入，Decoder 负责按顺序生成输出，所以 Decoder 必须遮住未来信息。

## 22. Decoder-Only 架构

现在很多大语言模型，包括 GPT 类模型，使用的是 Decoder-Only 架构。

经典 Transformer 是：

```text
Encoder：读输入
Decoder：生成输出
```

Decoder-Only 只保留并堆叠 Decoder 层：

```text
输入上下文 -> Decoder 层 -> 预测下一个 token
```

注意：Decoder-Only 不是“只有输出层”，而是只保留 Decoder 层。Decoder 层内部仍然有 Masked Self-Attention、FFN、Add & Norm 等结构。

Decoder-Only 的核心任务是自回归生成：

> 根据前面已经出现的 token，预测下一个 token。

例如：

```text
今天天气很 -> 好
今天天气很好 -> ，
今天天气很好， -> 适合
今天天气很好，适合 -> 出门
```

很多任务都可以统一成“给定上下文，续写下文”：

| 任务 | 统一成的形式 |
| --- | --- |
| 问答 | 问题 -> 答案 |
| 翻译 | 原文 -> 译文 |
| 总结 | 原文 -> 摘要 |
| 代码生成 | 需求 -> 代码 |
| 对话 | 历史消息 -> 下一轮回复 |

关键理解：

> Decoder-Only 模型把各种任务统一成“基于上下文预测下一个 token”的自回归生成问题。

## 23. Prompt Engineering 提示工程

提示工程不是简单把问题写长，而是：

> 用清晰、结构化、带约束的信息，引导模型产生更符合目标的输出。

因为 LLM 是根据上下文预测下一个 token，prompt 给出的上下文会显著影响输出。

弱提示：

```text
讲一下 Agent。
```

问题是太模糊，模型不知道：

```text
面向谁？
讲多深？
要不要举例？
要不要对比 Chatbot？
输出成文章、表格还是面试答案？
```

更好的提示：

```text
请用 AI 产品经理能听懂的语言解释 Agent。
要求：
1. 先用一句话定义；
2. 对比 Chatbot；
3. 用一个教育批改产品例子说明；
4. 最后给出 3 个常见误区。
```

好的 Prompt 通常包含：

```text
角色/受众
任务目标
输出结构
内容约束
示例方向
评价标准
```

关键理解：

> Prompt 是给模型的上下文设计；提示工程是在管理模型生成时的方向、边界和输出格式。

## 24. 角色、任务、格式与 Few-shot

一个 Prompt 通常可以拆成：

```text
你是谁 / 你扮演什么角色
你要完成什么任务
你要按什么格式输出
有什么限制或评价标准
```

例如：

```text
你是一名 AI 产品经理学习教练。
请帮我总结下面这篇文章，目标是让我能用于面试表达。
输出格式：
1. 一句话主旨
2. 三个核心概念
3. 一个产品例子
4. 两个容易混淆的点
要求：
- 不要超过 500 字
- 不要使用太多技术黑话
```

这里的结构是：

| Prompt 元素 | 作用 |
| --- | --- |
| 角色 | 让模型知道站在什么身份回答 |
| 任务 | 让模型知道要完成什么 |
| 格式 | 让输出结构更可控 |
| 约束 | 限制范围、风格、长度和边界 |

Few-shot Prompting 是在 Prompt 里给模型几个示例，让它模仿示例的模式完成新任务。

Zero-shot：

```text
请判断下面用户反馈的情绪是正面、负面还是中性：
“这个功能太难用了。”
```

Few-shot：

```text
请判断用户反馈情绪，只输出：正面 / 负面 / 中性。

示例1：
输入：这个功能很好用。
输出：正面

示例2：
输入：页面经常卡住。
输出：负面

示例3：
输入：我今天试了一下新版本。
输出：中性

现在判断：
输入：这个功能太难用了。
输出：
```

Few-shot 不是重新训练模型，因为它不会更新模型参数，只是在上下文里给模型一个可模仿的模式。

## 25. Chain-of-Thought 思维链提示

Chain-of-Thought Prompting，简称 CoT，中文常叫思维链提示。

它的核心是：

> 对复杂问题，不要只让模型直接给答案，而是引导它先按步骤推理，再得出结论。

例如产品需求分析：

```text
请按以下步骤分析这个产品需求是否值得做：
1. 目标用户是谁？
2. 用户痛点是否高频且强烈？
3. 现有替代方案是什么？
4. 我们的差异化在哪里？
5. MVP 可以怎么设计？
6. 主要风险是什么？
7. 最后给出结论。
```

CoT 和普通格式约束不同：

| 类型 | 核心 |
| --- | --- |
| 表格/列表格式 | 约束输出结构 |
| CoT | 约束推理过程 |

关键理解：

> 表格是输出结构，CoT 是推理结构。CoT 让模型沿着步骤走，减少直接拍脑袋给结论。

## 26. 提示工程的局限

Prompt 很有用，但不能替代系统设计。

提示工程无法从根本上解决：

| 局限 | 含义 |
| --- | --- |
| 知识缺口 | 模型不知道或知识过时，Prompt 不能凭空补知识 |
| 事实可靠性 | 模型可能生成看似合理但事实错误的内容 |
| 复杂任务稳定性 | 长任务、多步骤任务中，模型可能漏步骤、忘约束、跑偏 |
| 外部数据需求 | 有些问题必须查数据库、API、文件、网页或企业知识库 |

好的 AI 产品不能只靠 Prompt，还需要：

```text
清晰任务边界
外部工具 / RAG / 数据库
校验机制
反馈和纠错
失败兜底
状态、记忆和停止条件
```

关键理解：

> Prompt 能提高输出质量，但不能替代系统设计。

## 27. Tokenization 文本分词

LLM 不是直接处理完整句子，也不是一定按“词”处理，而是先用 tokenizer 把文本切成 token。

Token 可能是：

```text
一个英文词：agent
英文词的一部分：learn / ing
一个汉字：我
一个中文词或短语的一部分
一个标点：。
一个空格
特殊符号：<bos>、<eos>
```

所以：

```text
token != 单词
token != 句子
```

Tokenizer 通常负责：

```text
原始文本 -> token 序列 -> token ID 序列
```

随后模型会把 token ID 映射成 embedding 向量，再进入 Attention、FFN、Decoder 等模块。

完整链路：

```text
原始文本 -> token 序列 -> token ID 序列 -> embedding 向量 -> 模型处理
```

## 28. 子词分词与 BPE

不同模型可能使用不同 tokenizer。分词方式会影响 token 数量、处理效率和对词的表示方式。

按词分词：

```text
I love learning agents
-> I / love / learning / agents
```

问题：

- 词表会非常大；
- 遇到词表外新词，也就是 OOV，会很难处理。

按字符分词：

```text
learning
-> l / e / a / r / n / i / n / g
```

问题：

- 序列太长；
- 效率低；
- 每个 token 信息太碎。

现代 LLM 常用子词分词：

```text
learning -> learn / ing
unbelievable -> un / believe / able
```

它在“词表大小”和“处理未知词能力”之间做折中。

BPE，Byte Pair Encoding，直觉是：

> 从字符开始，反复把语料中最常一起出现的相邻片段合并成更大的 token。

例如：

```text
l o w
-> lo w
-> low
```

BPE 可以处理常见词，也可以处理没见过的新词：

- 常见片段会被合并成更大的 token；
- 不常见词仍然可以拆成字符或子词。

## 29. Token 数为什么重要

LLM 处理的是 token，不是“字数”或“词数”。

Token 数重要，主要有三个原因：

| 原因 | 含义 |
| --- | --- |
| 上下文窗口限制 | 模型一次能处理的 token 数有限，装不下就要截断、压缩或分块 |
| 成本 | 很多 API 按输入 token 和输出 token 计费 |
| 速度 | token 越多，计算量越大，响应通常越慢 |

长文档总结不能简单把全文塞给模型。

例如用户上传 100 页 PDF，更合理的做法是：

```text
分块 -> 摘要 -> 合并 -> 再总结
```

或者结合 RAG：

```text
先检索相关片段 -> 再让模型回答
```

关键理解：

> Token 数决定了模型能看多少、花多少钱、跑多快。

## 30. 调用开源大语言模型

如果不用在线 API，而是调用开源模型，基本流程是：

```text
选择模型
-> 加载 tokenizer
-> 加载模型
-> 准备输入
-> 编码成 token ID
-> 模型生成
-> 解码成文本
```

每一步的意义：

| 步骤 | 作用 |
| --- | --- |
| 选择模型 | 决定用哪个开源 LLM，例如 Qwen、Llama、ChatGLM 等 |
| 加载 tokenizer | 用模型对应的分词器切文本、转 token ID |
| 加载模型 | 把模型权重加载到 CPU/GPU 上 |
| 准备输入 | 写 prompt 或对话消息 |
| 编码输入 | 把文本转成模型能处理的 token ID |
| 模型生成 | 根据输入预测后续 token |
| 解码输出 | 把生成的 token ID 转回可读文本 |

Tokenizer 必须和模型匹配。因为不同模型的词表、token ID 映射和特殊符号可能不同。

例如：

```text
同一个 ID 12345
在 A 模型词表里可能代表 "Agent"
在 B 模型词表里可能代表另一个片段
```

关键理解：

> 调用开源 LLM 的基本流程是：文本进来先编码，模型生成 token，最后再解码成人能读的文字。

## 31. 模型加载、设备与生成参数

开源模型调用通常会涉及设备选择：

```text
device = "cuda" if torch.cuda.is_available() else "cpu"
```

意思是：

> 如果有可用 GPU，就优先用 GPU，否则用 CPU。

因为 LLM 推理本质上是大量矩阵计算，GPU 更擅长并行计算，通常比 CPU 快很多。

模型权重可以理解成：

> 模型训练后学到的参数，也就是能力载体。

开源模型不是越大越适合本地跑，还要考虑：

```text
显存/内存是否够
推理速度是否可接受
模型许可证是否允许商用
任务是否真的需要这么大的模型
```

常见生成参数：

| 参数 | 作用 |
| --- | --- |
| `max_new_tokens` | 控制模型最多生成多少个新 token |
| `temperature` | 控制生成随机性，越高越发散，越低越稳定 |
| `top_p` | 从累计概率达到 p 的候选 token 集合中采样 |

产品直觉：

| 场景 | 参数倾向 |
| --- | --- |
| 事实问答、法律、医学、代码修复 | 低 temperature，更稳 |
| 文案、创意、头脑风暴 | 稍高 temperature，更发散 |
| 摘要、分类、结构化抽取 | 低 temperature，输出更一致 |
| 长文生成 | 控制 max_new_tokens，避免超长或截断 |

注意：

> 低 temperature 只是减少随机性，不代表自动保证事实正确。

## 32. 模型选择

模型选择不能只看“哪个模型最强”，而要综合考虑：

```text
能力
成本
速度
上下文长度
部署方式
隐私与安全
许可证
生态和工具支持
```

不同任务的选择倾向：

| 任务 | 模型选择倾向 |
| --- | --- |
| 简单分类、打标签、格式转换 | 小模型可能够用 |
| 复杂推理、多轮规划、代码生成 | 需要更强模型 |
| 长文档问答 | 需要长上下文或 RAG |
| 企业内部数据 | 更关注私有化部署、权限和数据安全 |
| 高并发在线产品 | 成本和延迟很关键 |
| 商业产品 | 许可证和商用限制必须检查 |

关键理解：

> 模型选择不是追求参数最大，而是选择刚好能稳定完成任务的模型。

## 33. Scaling Law 缩放法则

缩放法则讲的是模型训练阶段的规律，不是用户对话时发生的事。

它关注三类训练资源：

```text
参数量
训练数据量
计算量
```

在合理配比下，增加这三类资源，模型性能通常会提升。

注意它不是说用户写更好的 Prompt，就能让小模型临时变成大模型。Prompt 可以让模型把已有能力发挥得更好，但不能改变模型参数量、训练数据量和训练时用过的计算量。

可以这样区分：

| 概念 | 发生阶段 | 讲的是什么 |
| --- | --- | --- |
| 自回归生成 | 使用/推理阶段 | 模型如何一个 token 一个 token 生成文本 |
| Prompt Engineering | 使用/推理阶段 | 用户如何设计上下文影响输出 |
| 缩放法则 | 训练/研发阶段 | 参数量、数据量、计算量和能力提升之间的规律 |

为什么不能只增加参数量？

> 模型容量变大了，但如果训练数据不够、计算量不够，它就学不充分，可能浪费容量，甚至效果不如合理配比的小模型。

产品经理视角：

> 更强模型往往来自更大训练投入，但也意味着更贵、更慢、更难部署。模型选型要看任务收益是否值得这些成本。

## 34. 模型幻觉 Hallucination

模型幻觉指的是：

> 模型生成了看起来合理、语气自信，但事实上错误或没有依据的内容。

例子：

```text
编造不存在的论文
编造错误的法律条文
把 API 用法说错
虚构数据来源
错误总结文档
```

为什么会幻觉？

> LLM 的核心机制是基于已有上下文预测下一个 token。它优化的是生成符合语言模式、看起来合理的文本，不等于每句话都经过事实数据库校验。

所以模型可能：

```text
语言很流畅
结构很完整
语气很确定
但事实是错的
```

应对幻觉的方法：

```text
让模型引用来源
接入搜索、数据库、RAG
对关键结果做规则校验
显示置信度或不确定性
设置人工复核
限制模型在高风险场景自由发挥
```

这些方法只能降低幻觉风险，不能保证完全消灭幻觉。

关键理解：

> 幻觉不是模型故意骗人，而是生成式语言模型在缺少事实约束时，把合理语言模式误当成可靠事实输出。

## 35. 第三章总线索

第三章可以串成一条主线：

```text
语言模型从统计方法开始
-> N-gram 用局部统计近似语言概率
-> 神经网络语言模型用词向量解决语义表示问题
-> RNN/LSTM 按顺序传递上下文
-> Transformer 用注意力机制提升长距离依赖和并行能力
-> Decoder-Only 统一成基于上下文预测下一个 token
-> Prompt、Tokenizer、模型调用和模型选择决定我们如何使用 LLM
-> Scaling Law 解释能力提升的一部分来源
-> 幻觉提醒我们 LLM 仍然需要事实约束和系统设计
```

从 `N-gram -> RNN/LSTM -> Transformer` 的演进：

| 阶段 | 主要解决的问题 | 新问题或局限 |
| --- | --- | --- |
| N-gram | 用局部统计估计语言概率 | 数据稀疏、泛化差、缺少语义表示 |
| 词向量/神经网络语言模型 | 让词有连续语义表示，能学习相似关系 | 还需要更好处理上下文顺序 |
| RNN/LSTM | 按顺序传递上下文，突破固定窗口 | 长距离依赖仍难，并行效率低 |
| Transformer | 用注意力机制直接建立 token 关系，并提升并行效率 | 仍受上下文窗口、成本、幻觉等限制 |

第三章最重要的理解：

> LLM 不是魔法黑箱，而是从“预测下一个 token”这个基础任务出发，通过更强的表示、更大的规模、更好的架构，逐步形成了今天的通用生成能力。

从 AI 产品经理视角，这章有三个落点：

1. 理解能力来源：LLM 的输出来自上下文预测、训练数据、模型参数和架构设计，不是查标准答案。
2. 理解使用边界：Prompt、分词、上下文窗口、生成参数、模型选择都会影响效果、成本和稳定性。
3. 理解产品治理：幻觉、过时知识、复杂任务偏离不能只靠提示词解决，需要工具、RAG、校验、权限、人工复核和兜底。

## 36. 今日问答纠偏

今天重点纠正了几个理解：

- 语言模型不是查数据库拿标准答案，而是在上下文中预测 token，因此可能有随机性和幻觉。
- 链式法则中的 `|` 表示 given，也就是在某些条件已经发生时计算概率。
- Bigram 统计的是相邻且有顺序的词对，不能说成简单“同时出现”。
- `Count(agent works)` 和 `Count(works agent)` 不一样，因为顺序不同。
- N-gram 的两个核心缺陷是数据稀疏性和泛化能力差。
- 词向量的泛化不是显式查同义词，而是通过相似语义表示自然迁移。
- 余弦相似度主要比较向量方向是否接近，不是简单比较词面距离。
- RNN 的隐藏状态是上下文摘要，不是每一步重新读完整前文。
- Transformer 可以在上下文窗口内直接建立 token 关系，但不是无限全局理解。
- Encoder 的“理解”是向量表示和模式计算，不是人类意识意义上的理解。
- Token 不一定等于一个词。中文短语可能被切成多个 token。
- 每个 token 都有 Q、K、V，不是不同 token 分别只承担一种身份。
- Q/K 匹配产生的是原始相关性分数，归一化后才是注意力权重。
- 权重乘以 V 的目的，是决定从对应 token 的 Value 中汇总多少信息。
- `d_model=512` 表示每个 token 的内部表示是 512 维向量，不是 binary，也不是平方。
- 单头注意力不是看不到全句，而是只有一个观察角度；多头注意力用多个表示子空间看同一批 token。
- 多头里“一个头看指代、一个头看语法”只是教学类比，不是工程师手动写死规则。
- 多个 head 算完后要拼接并线性变换，是为了把不同子空间里的观察结果融合成统一表示。
- Add & Norm 是训练稳定结构，不是语言理解本身。
- FFN 是对每个 token 分别做非线性加工，不是直接比较 token 与 token。
- Decoder 要加 mask，是为了防止预测当前位置或下一个 token 时偷看未来 token。
- Decoder-Only 不是只有输出层，而是只保留并堆叠 Decoder 层。
- 表格是输出结构，CoT 是推理结构。
- Prompt 可以改善输出，但不能替代系统设计。
- Tokenizer 主要负责文本到 token、token 到 ID；embedding 是模型后续做的。
- 低 temperature 只是降低随机性，不保证事实正确。
- 缩放法则发生在训练阶段，不是用户对话时让模型自己变大。
- 词向量/神经网络主要解决 N-gram 的语义表示和泛化问题；RNN/LSTM 解决固定窗口问题；Transformer 解决 RNN/LSTM 的长距离依赖和并行效率问题。
