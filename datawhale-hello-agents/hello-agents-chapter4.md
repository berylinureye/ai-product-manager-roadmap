## 1. 本章主题和学习目标

第四章主题是智能体经典范式构建。前三章主要帮助理解智能体、大语言模型和工具调用的基础，第四章开始进入更具体的 Agent 构建方法。

本章核心问题：

> 如何把 LLM 的推理能力、外部工具、执行循环和反思机制组合成可运行的智能体系统？

本章重点学习三种经典范式：

- `ReAct`: 边推理边行动，通过 `Thought -> Action -> Observation` 循环完成任务。
- `Plan-and-Solve`: 先生成整体计划，再逐步执行，最后汇总答案。
- `Reflection`: 先生成初始答案，再批判、修正，提高输出质量。

关键理解：

> Agent 不只是一个聊天模型，而是一个由 LLM、工具、提示词协议、解析器、执行器、历史状态和循环控制共同组成的系统。

## 2. 为什么要自己实现经典范式

虽然 LangChain、LlamaIndex 等框架已经封装了很多 Agent 能力，但第四章仍然选择手写基础范式。

原因有三点：

- 只使用高层框架，容易只知道如何调用 API，但不知道 Agent 内部如何组织 prompt、解析输出、调用工具和维护状态。
- 自己实现会暴露真实工程问题，例如输出格式不稳定、工具调用失败、循环不结束、历史过长等。
- 理解底层范式后，才能在复杂业务中判断该改哪里、为什么改、改了会影响什么。

产品经理视角：

> 不一定每天写底层代码，但要理解系统结构和边界。这样在设计 Agent 产品、排查问题、评审方案时，才能问到关键问题。

## 3. 环境准备与基础 LLM 调用

第四章先搭基础设施，再进入具体范式。

常用依赖：

```bash
pip install openai python-dotenv
```

两个库的作用：

- `openai`: 用来调用兼容 OpenAI 接口的大语言模型服务，不一定只能调用 OpenAI 官方模型。
- `python-dotenv`: 用来读取 `.env` 文件中的配置。

`.env` 文件通常存放：

```bash
LLM_API_KEY="YOUR-API-KEY"
LLM_MODEL_ID="YOUR-MODEL"
LLM_BASE_URL="YOUR-URL"
```

字段含义：

- `LLM_API_KEY`: 模型服务访问凭证。
- `LLM_MODEL_ID`: 要调用的具体模型。
- `LLM_BASE_URL`: 模型服务地址。

`.env` 和 `.gitignore` 的区别：

- `.env`: 真正存放密钥和配置。
- `.gitignore`: 告诉 Git 不要提交 `.env`，防止密钥泄露。

关键理解：

> 配置和代码分离，可以让同一套 Agent 逻辑切换不同模型、不同服务商和不同环境。

基础 LLM 调用被封装成 `HelloAgentsLLM`：

- `__init__`: 初始化模型名、API key、base URL、超时时间等配置。
- `think(messages, temperature=0)`: 接收上下文消息，调用模型并返回生成文本。

封装的价值：

> ReAct、Plan-and-Solve、Reflection 都需要调用 LLM。把模型调用统一封装后，后面的 Agent 范式只需要关心什么时候调用、传什么上下文、如何处理回复。

## 4. ReAct

`ReAct` 是 `Reason + Act`，核心是让模型在推理和行动之间循环。

基本轨迹：

```text
Thought -> Action -> Observation -> Thought -> Action -> Observation -> ... -> Final Answer
```

三个核心字段：

- `Thought`: 模型分析当前情况，决定下一步。
- `Action`: 模型选择要调用的工具和输入参数。
- `Observation`: 工具执行后返回的结果，写回历史，成为下一轮思考的依据。

关键理解：

> ReAct 不是让模型一次性想完，而是让思考指导行动，行动结果再反过来修正思考。

### 4.1 ReAct 解决的问题

纯思考方法，例如 Chain-of-Thought，只能在模型内部推理，不能接触外部世界。遇到最新事实或外部数据时，容易凭记忆编错。

纯行动方法可以直接调用工具，但缺少明确的规划、推理和纠错过程。工具选错或参数传错时，系统不一定能修正。

ReAct 把两者结合起来：

- 思考让行动更有目的。
- 行动让模型接触外部世界。
- 观察结果让下一轮推理有真实反馈。

### 4.2 工具定义与 SearchTool

ReAct 需要工具来完成行动。原文先实现搜索工具 `SearchTool`，用于调用 Serper API 获取外部搜索结果。

工具不只是普通函数，而要有：

- `name`: 工具名，例如 `search`，用于 Action 中引用。
- `description`: 工具说明，帮助模型判断什么时候该用这个工具。
- 输入格式: 告诉模型应该传什么参数。
- 输出格式: 返回模型容易理解的 Observation。

搜索工具的输出不应该直接把原始 JSON 全部丢给模型，而应提取标题、链接、摘要等关键信息，格式化成清晰文本。

关键理解：

> 工具输出要为 Agent 的下一轮思考服务，而不是把所有原始数据无差别塞回上下文。

### 4.3 ToolExecutor

`ToolExecutor` 是工具调度器，不负责具体搜索或计算。

它负责：

- 接收工具列表。
- 注册成 `{工具名: 工具对象}` 的映射。
- 根据模型输出的工具名找到对应工具。
- 把工具输入传给工具的 `run()` 方法。
- 返回执行结果或错误信息。

示例：

```text
Action: search[华为最新手机]
```

执行器会解析出：

```text
tool_name = "search"
tool_input = "华为最新手机"
```

然后执行：

```python
SearchTool.run("华为最新手机")
```

关键理解：

> LLM 负责选择工具和生成参数；ToolExecutor 负责按名称找到真实工具并执行。

### 4.4 系统提示词与输出协议

ReAct 依赖系统提示词规定输出格式：

```text
Thought: 你对当前任务的思考
Action: 工具名称[工具输入]
Observation: 工具执行结果
...
Thought: 我现在知道最终答案了
Final Answer: 最终答案
```

系统提示词还会插入可用工具列表，例如：

```text
search: 当你需要回答关于时事、最新事件或需要实时信息的问题时很有用。
```

关键理解：

> 系统提示词把 LLM 的自由生成约束成一个可执行协议。程序依靠 `Action:` 和 `Final Answer:` 等结构化标记决定下一步。

### 4.5 核心循环

ReActAgent 的 `run()` 不是一次模型调用，而是循环控制器。

核心流程：

```text
用户问题 + 历史
-> LLM 输出 Thought/Action 或 Final Answer
-> Parser 解析 Action 或 Final Answer
-> ToolExecutor 执行工具
-> Observation 写回历史
-> 下一轮 LLM 调用
```

如果出现 `Final Answer`，循环结束。

如果出现 `Action`，程序解析工具名和参数，调用工具，然后把结果作为 `Observation` 追加回历史。

必须设置 `max_iterations`，防止模型一直搜索、一直行动、迟迟不给最终答案。

### 4.6 输出解析器

输出解析器连接自然语言和程序执行。

常见解析方法：

- `_parse_action(response)`: 从模型输出中提取工具名和工具输入。
- `_parse_final_answer(response)`: 判断是否已经输出最终答案。

如果模型没有严格按格式输出，解析器可能失败。工程上可以：

- 增强解析器，兼容轻微格式偏差。
- 把格式错误作为 Observation 或反馈交给模型，让它按格式重试。
- 受 `max_iterations` 控制，避免无限重试。

### 4.7 ReAct 的特点和局限

主要特点：

- 可解释性强：能看到 `Thought / Action / Observation` 轨迹，方便判断是哪一步出错。
- 灵活性高：可以接入搜索、计算器、数据库、API、代码执行器等工具。
- 鲁棒性较好：工具失败或结果不理想时，有机会在下一轮调整。

固有局限：

- 对提示词和输出格式敏感。
- 可能陷入循环，需要 `max_iterations`。
- 成本较高，因为多轮 LLM 调用、多次工具调用和不断增长的历史都会增加 token、延迟和 API 成本。

调试技巧：

- 仔细设计工具描述。
- 记录完整 `Thought / Action / Observation` 轨迹。
- 设置最大迭代次数。
- 工具调用失败时优雅处理，把错误反馈给模型或退出，而不是直接崩溃。

适用任务：

- 需要最新信息。
- 需要工具调用。
- 需要边查边判断。
- 例如查询新闻、排查问题、调用 API 完成任务。

## 5. Plan-and-Solve

`Plan-and-Solve` 的核心是：

```text
先规划，再执行。
```

它把复杂任务拆成两个阶段：

- `Plan`: 先生成完整解决路径。
- `Solve`: 按计划逐步执行，并记录每一步结果。

适合任务：

- 复杂、多步骤、需要整体结构的任务。
- 例如学习计划、研究报告、方案设计、长文写作、市场进入策略。

关键理解：

> ReAct 更像探索，Plan-and-Solve 更像项目管理。

### 5.1 工作原理

Plan 阶段输出的是计划步骤，不是最终答案。

Solve 阶段每一步执行时，需要参考：

- 原始问题 `question`
- 完整计划 `plan`
- 已完成步骤和结果 `history`
- 当前步骤 `current_step`

这样可以保证当前输出服务整体目标，不重复已完成内容，也不跳到后面步骤。

### 5.2 规划阶段

规划阶段通过 `planner_prompt` 要求模型：

- 扮演擅长拆解复杂问题的规划专家。
- 把问题拆成清晰、具体、可执行的步骤。
- 不要直接回答问题。
- 只输出计划。

好的计划应该：

- 覆盖关键环节。
- 步骤清晰具体。
- 可以执行。
- 不碎成一堆无意义的小动作。

如果计划太粗，后面的执行会普通；如果计划太碎，会增加 token 成本，也会限制执行弹性。

### 5.3 执行器与状态管理

执行阶段需要维护状态：

```text
question: 原始问题
plan: 完整计划
history: 已完成步骤及其结果
current_step: 当前步骤
```

`history` 在 Plan-and-Solve 中记录的是每个计划步骤的执行结果。

它的作用：

- 避免重复前面步骤。
- 保持前后结论一致。
- 让当前步骤接着已有结果继续推进。

### 5.4 单步执行 prompt

`solver_prompt` 会包含：

```text
原始问题:
{question}

完整计划:
{plan}

历史步骤与结果:
{history}

当前步骤:
{current_step}
```

并要求模型只完成当前步骤，不要跳到后面步骤，也不要直接输出最终答案。

关键理解：

> 给足上下文，但收窄当前任务范围。

给足上下文是为了不跑偏、不重复、不和前文冲突；收窄范围是为了不抢跑、不提前完成后续步骤。

### 5.5 最终汇总

每一步执行结果只是局部产出，不能简单拼起来当最终答案。

最终需要把：

- 原始问题
- 完整计划
- 每一步执行结果

交给 LLM 做整合，生成连贯、有结构、能回答原问题的最终答案。

完整链路：

```text
Plan -> Solve step by step -> Synthesize final answer
```

### 5.6 Plan-and-Solve 的特点和局限

优势：

- 先建立全局结构，不容易遗漏关键环节。
- 适合多步骤、结构化任务。
- 输出更像一个完整方案，而不是想到哪写到哪。

局限：

- 强依赖初始计划质量。
- 如果 Plan 阶段漏掉关键环节，Solve 阶段可能会沿着错误路径执行。
- 计划可能把后续执行框住，灵活性弱于 ReAct。

真实系统中可以混合使用：

```text
先用 Plan-and-Solve 做整体计划
-> 在需要外部资料或不确定信息的步骤中用 ReAct
-> 最后汇总成完整答案
```

## 6. Reflection

`Reflection` 的核心是：

```text
Initial Answer -> Critique / Reflection -> Revised Answer
```

它让 Agent 不只做生成者，还做评估者和优化者。

三个角色：

- `Generator`: 生成初始答案或初始代码。
- `Reflector`: 审查初始结果，指出问题、遗漏、逻辑错误、边界情况等。
- `Refiner`: 根据反思意见修改初始结果，生成更好的版本。

这些角色可以由同一个 LLM 在不同 prompt 下分阶段扮演，也可以由不同模型承担。

关键理解：

> Reflection 不是简单再问模型一次，而是用不同角色和不同提示词组织一次质量改进循环。

### 6.1 Reflection 为什么有用

生成和评估是两种不同任务。

第一次生成时，模型主要关注产出答案，容易忽略细节错误、约束条件或边界情况。

反思阶段会把任务切换成找问题，例如：

- 是否满足原始需求？
- 有没有遗漏关键条件？
- 有没有逻辑矛盾？
- 有没有边界情况？
- 有没有更好的表达或实现？

所以 Reflection 不是让模型突然变聪明，而是通过角色切换和任务重构，让模型把注意力放在不同维度上。

### 6.2 代码案例与 Memory

原文用代码生成做 Reflection 案例，例如：

```text
编写一个 Python 函数，计算列表中所有偶数的平方和。
```

代码生成适合 Reflection，因为：

- 需求明确。
- 容易出现 bug。
- 可以检查边界情况。
- 修改前后容易比较。
- 反思意见比较具体。

这里的 `Memory` 是一个轻量记录模块，不是复杂的长期记忆或向量数据库。

它主要保存：

- 原始任务。
- 每一轮生成的代码。
- 每一轮反思意见。
- 每一轮优化后的代码。

常见方法：

- `Memory.add(entry)`: 保存本轮信息。
- `Memory.get_history()`: 取出历史记录，放入后续 prompt。

关键理解：

> Memory 给反思循环提供上下文连续性，避免模型忘记上一轮为什么修改、哪里还没有解决。

### 6.3 Reflection 的提示词设计

原文设计两个关键 prompt：

`reflection_prompt`:

- 接收原始任务和待审查代码。
- 要求模型作为代码审查员找问题。
- 输出具体反思意见。
- 不负责直接写新代码。

`refinement_prompt`:

- 接收原始任务、上一轮代码和反思意见。
- 根据反思意见输出改进后的代码。

角色拆开的价值：

> 每次模型调用的目标更清晰，输出更可控。Reflector 专心找问题，Refiner 专心根据问题修正。

### 6.4 ReflectionAgent

`ReflectionAgent` 通常包含：

- `generate_initial_solution(task)`: 生成初始答案。
- `reflect(task, solution)`: 生成反思意见。
- `refine(task, solution, reflection)`: 根据反思意见生成改进版本。
- `run(task, max_iterations)`: 控制生成、反思、优化、记录和多轮迭代。

整体流程：

```text
Initial solution
-> Reflection feedback
-> Refined solution
-> Memory.add(...)
-> 下一轮继续 reflect/refine
-> 达到迭代次数后输出最终版本
```

Reflection 也需要 `max_iterations`，因为无限反思会增加成本和延迟，还可能越改越复杂、越改越偏。

### 6.5 成本收益分析

主要成本：

- 计算成本增加：初始生成、反思、优化都需要模型调用。
- 延迟增加：用户等待时间更长。
- 可能过度优化：简单答案可能被改得复杂，甚至引入新问题。

主要收益：

- 提高输出质量。
- 增强鲁棒性。
- 提供可解释的改进过程。

适合任务：

- 代码生成和代码审查。
- 长文写作。
- 产品方案评审。
- PRD 检查与优化。
- 合同草案、上线方案等高价值或高风险任务。

不适合任务：

- 简单、低风险、需要快速回答的问题。
- 例如“Python 列表怎么取第一个元素”。

## 7. 三种范式的对比

| 范式 | 核心机制 | 适合任务 | 主要优势 | 主要局限 |
| --- | --- | --- | --- | --- |
| ReAct | 边推理边行动，工具结果回填历史 | 最新信息、工具调用、排查问题、API 操作 | 灵活、可解释、能接入外部工具 | 格式敏感、可能循环、成本高 |
| Plan-and-Solve | 先规划，再逐步执行，最后汇总 | 学习计划、研究报告、方案设计、长文写作 | 结构清晰、不易漏步骤 | 依赖初始计划质量，灵活性较弱 |
| Reflection | 先生成，再批判，再修正 | 代码、PRD、方案评审、高质量写作 | 提高质量、减少明显错误、过程可解释 | 成本和延迟增加，可能过度优化 |

一句话判断：

- 任务不确定、需要工具反馈：优先考虑 `ReAct`。
- 任务复杂、需要整体结构：优先考虑 `Plan-and-Solve`。
- 输出质量重要、需要审查修订：优先考虑 `Reflection`。

真实产品中三者常常组合使用。

例如市场进入策略：

```text
Plan-and-Solve: 先规划市场、用户、竞品、法规、渠道、风险
ReAct: 在竞品、法规、市场数据等步骤中调用搜索工具
Reflection: 最后审查策略是否遗漏关键风险、结论是否自洽
```

## 8. 我的学习轨迹：从“会用 Agent”到“能判断范式”

第四章最容易写成框架说明书，但学习过程中真正有价值的地方，是我开始把 Agent 看成一个有职责分工、有执行边界、需要调试的系统。

### 8.1 第一层理解：LLM 是大脑，Agent 是带执行链路的系统

我一开始对第四章的开场理解是：

```text
普通聊天模型：主要生成回答
Agent：理解任务 -> 思考/拆解 -> 执行 -> 观察结果 -> 再思考 -> 再执行
```

也就是说，Agent 比 LLM 多了外部工具、执行链路、观察反馈和循环控制。

这也解释了为什么 Agent 风险更多：不是模型变坏了，而是系统环节变多了。工具可能选错，参数可能传错，工具结果可能过时或不可靠，模型也可能误读 Observation。

所以第四章对我的第一条产品启发是：

> Agent 产品不是“给 LLM 一个更长 prompt”就结束，而是要设计任务流、工具权限、错误处理、状态记录和停止边界。

### 8.2 第二层理解：框架不是黑箱，手写范式是为了看见内部机制

学到 LangChain 和 LlamaIndex 时，我一开始并不知道它们是什么，还问要不要跳过。

后来只保留了最小必要理解：

```text
LangChain / LlamaIndex 代表成熟框架和现成封装
第四章手写 ReAct、Plan-and-Solve、Reflection
是为了看懂框架背后的 prompt、parser、executor、history 和 loop
```

这个点对产品经理也有意义。会用一个框架，不等于知道 Agent 为什么会出错；理解底层范式，才能在评审技术方案时问出更具体的问题：

```text
工具描述是否足够清楚？
模型输出格式错了怎么办？
循环不结束怎么办？
历史太长怎么办？
工具失败怎么反馈给模型？
什么时候应该转人工或停止？
```

### 8.3 第三层理解：代码不需要背，但要看懂职责

学到安装依赖、`.env`、API key 时，我问过需不需要跟着实操安装。

最后确定的学习方式是：

> 代码要读懂意图，不需要背。第四章代码是在拆 Agent 的内部骨架，而不是要求我记住每一行实现。

这里有一个具体纠偏：我曾把 `.env` 和 `.gitignore` 混在一起。

最后区分为：

```text
.env：真正存放 API key、模型名、服务地址等配置
.gitignore：告诉 Git 不要提交 .env，避免密钥泄露
```

配置和代码分离的价值，不只是安全，也是产品灵活性：开发环境可以用便宜模型，线上环境用稳定模型；同一套 Agent 逻辑也可以切换不同服务商或本地模型。

### 8.4 ReAct：不是“搜一下”，而是让行动结果进入下一轮思考

我对 ReAct 的初始理解是“边想边做”，后来变得更具体：

```text
Thought：模型判断下一步要做什么
Action：选择工具并生成输入
Observation：工具返回结果
下一轮 Thought 基于 Observation 继续调整
```

所以 ReAct 的关键不是“会调用搜索”，而是工具结果会被写回历史，成为下一轮推理依据。

这也解释了为什么它需要：

- 工具描述：让模型知道什么时候用哪个工具；
- ToolExecutor：把模型的文本 Action 变成真实工具调用；
- Parser：从自然语言输出里解析工具名、参数或 Final Answer；
- `max_iterations`：防止一直查、一直想、不结束。

我当时形成的判断是：

> ReAct 适合任务不确定、需要外部信息、需要边查边判断的场景，比如最新新闻、资料检索、排查问题、API 操作。

### 8.5 Plan-and-Solve：更像项目管理

Plan-and-Solve 让我把复杂任务理解成“先全局拆解，再逐步执行”。

我最后给自己的类比是：

> ReAct 更像探索，Plan-and-Solve 更像项目管理。

它的核心不是直接回答，而是先把问题拆成计划，再让每一步执行时都看到：

```text
原始问题
完整计划
已完成步骤和结果
当前步骤
```

这样做的价值是：当前步骤不会重复前面内容，也不容易提前跳到后面步骤。历史在这里记录的是“步骤执行结果”，不是简单聊天记录。

我的产品判断是：

> 学习计划、研究报告、方案设计、长文写作这类任务，需要整体结构，优先考虑 Plan-and-Solve。

### 8.6 Reflection：角色拆开，是为了让输出更可控

Reflection 是我这一章理解最细的一段。

我最后把它理解成三种角色：

```text
Generator：先生成初始答案或代码
Reflector：只负责审查和找问题
Refiner：根据反思意见修订结果
```

这里关键不是“再问一次模型”，而是把生成、审查、修订拆成不同目标。

我当时解释过：如果 `reflection_prompt` 一边审查一边直接改代码，工作量会变大，输出格式也容易混乱；而 `refinement_prompt` 带上反思意见，是为了让修订基于明确反馈，而不是重新泛泛生成。

这让我形成了一个更工程化的判断：

> 角色拆开，是为了让每次模型调用目标更清晰，输出更可控。

Reflection 也不是越反思越好。它需要 `max_iterations`，因为多轮反思会增加成本和延迟，还可能越改越复杂、越改越偏。

### 8.7 Reflector prompt 不是越挑剔越好

在“计算列表中所有偶数的平方和”的例子里，我意识到 Reflection 可以从逻辑、非数字元素、可读性、鲁棒性、简洁性等角度改进代码。

但这里也有风险：如果 Reflector 过度追求某一个标准，比如把鲁棒性放到极端，简单函数可能被改得很臃肿。

所以设计 Reflector prompt 时，要说清楚：

```text
优先检查什么问题
在什么前提下改进
哪些复杂化是不必要的
简洁性、鲁棒性、可读性之间如何取舍
```

这对 AI 产品也一样：评审标准如果不清楚，Agent 可能会把输出朝错误方向优化。

### 8.8 最后的范式选择纠偏

最后综合题里，我判断“最新融资新闻”适合 ReAct，“8 周 AI 产品经理学习计划”适合 Plan-and-Solve，这两个方向是对的。

但我把“PRD 挑问题并改进”说成了 ReAct。后来纠正为：

```text
如果只是审查和改进已有 PRD：优先 Reflection
如果评审 PRD 时还需要查竞品、法规、市场数据：Reflection + ReAct 混合
```

这次纠偏让我对三种范式形成了更实用的判断表：

| 场景问题 | 更适合的范式 |
| --- | --- |
| 需要最新信息、工具反馈、边查边判断 | ReAct |
| 任务复杂，需要先拆结构、逐步完成 | Plan-and-Solve |
| 已有初稿，需要审查、修订、提高质量 | Reflection |
| 真实复杂业务 | 多种范式组合 |

## 9. 本章 AI 产品经理视角总结

本章对 AI 产品经理最重要的启发：

1. Agent 的能力不是只靠一个大 prompt，而是由 LLM、工具、协议、解析器、执行器、状态和循环共同组成。
2. 工具描述是 Agent 能否正确行动的关键，描述模糊会导致不用工具或乱用工具。
3. 历史状态很重要。ReAct 记录工具反馈，Plan-and-Solve 记录步骤结果，Reflection 记录版本和反思意见。
4. 循环都需要边界。无论 ReAct 还是 Reflection，都要设置最大迭代次数，避免无限循环、成本失控或越改越乱。
5. 不同任务要选不同范式，真实产品里经常混合使用，而不是执着于单一范式。

对产品设计的启发：

- 如果做客服 Agent，需要把订单查询、知识库搜索、退款工单、转人工等业务能力封装成工具。
- 如果做研究或方案类 Agent，需要先规划结构，再在需要外部信息的步骤调用工具。
- 如果做高质量交付类 Agent，需要加入反思、审查、修订和人工确认机制。

最终理解：

> LLM 负责理解和推理，工具负责真实执行，Prompt 定义行为协议，Parser 把文本变成可执行指令，Executor 调用真实能力，History/Memory 保证模型不是失忆状态，Iteration 控制系统边界。
