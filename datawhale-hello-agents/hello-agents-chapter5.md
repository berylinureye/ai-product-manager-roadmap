## 1. 本章主题和学习目标

第五章主题是基于低代码平台的智能体搭建。

第四章通过 Python 代码理解了 ReAct、Plan-and-Solve、Reflection 等经典 Agent 范式。第五章开始把视角从“自己写底层实现”切到“用平台更快搭建、调试、发布 Agent”。

本章核心问题：

> 当我们已经理解 Agent 底层机制后，如何用低代码平台更快把 Agent 落到真实业务场景里？

本章按原文顺序学习了四类平台：

- `Coze`: 低门槛、快速创建和发布 Agent。
- `Dify`: 开源、可部署、可扩展的 LLM 应用开发与运营平台。
- `FastGPT`: 以知识库问答和 RAG 为核心的平台。
- `n8n`: 工作流自动化平台，适合把 AI 接入业务流程。

关键理解：

> 低代码平台不是让产品经理不懂技术，而是把重复工程能力平台化，让产品经理能更快验证业务逻辑，同时仍然要理解平台封装了什么、黑盒在哪里、出了问题该从哪里排查。

## 2. 我的学习轨迹总览

这一章不是简单看平台介绍，而是在不断纠偏中形成平台选型判断。

最开始我容易把低代码理解成：

> Agent 已经设计好了，只要知道怎么用就好了。

后来纠正为：

> 低代码平台只是把 API 调用、状态管理、工具接入、节点编排、日志调试等底层工程能力产品化。产品经理仍然要理解技术边界、数据流、工具调用、知识库召回、输出质量和排查路径。

另一个重要卡点是 Coze。

我亲自体验新版 Coze 后发现，它和教材截图已经不一样。教材里更像可视化搭积木，新版更像通过对话创建 AI 应用。这让我产生一个很重要的判断：

> 平台越自动化、越对话式，创建门槛越低，但黑盒感可能越强。

最后我形成的平台选型框架是：

> 不是问哪个平台最好，而是问当前任务的主语是什么：是快速发布 Agent、构建可控 LLM 应用、做 RAG 知识库问答，还是自动化业务流程。

## 3. 5.1 平台化构建的兴起

第五章开头强调：第四章“重复造轮子”对学习很重要，但真实工程里不能永远从零写代码。

低代码平台出现，是为了把常见底层能力封装成节点、模块、插件和工作流，让团队更快关注业务逻辑。

低代码平台的核心价值：

- 降低技术门槛：把 API 调用、状态管理、并发控制等封装成节点或模块。
- 提升开发效率：适合快速验证原型，而不是先花大量时间搭底层工程。
- 提供可视化与可观测性：能看到数据如何在节点间流动、哪个工具失败、哪个环节耗时。
- 标准化与最佳实践沉淀：提供模板、组件、工具接入规范，降低团队协作成本。

### 我的关键纠偏

我一开始把“实现细节”主要理解成 prompt 设计。后来纠正为：prompt 只是其中一部分，低代码平台更常封装的是 API 调用、状态管理、工具调用、节点连接、并发控制、日志和失败处理。

我也曾说低代码平台里的 Agent 已经设计好了，只要知道怎么运用就行。这个理解被纠正为：

> 平台封装了底层工程能力，但产品经理仍然要能 debug。回答质量差时，要判断是提示词问题、知识库召回问题、工具配置问题、流程路由问题，还是模型能力问题。

### 最终理解

低代码不是降低思考要求，而是把代码能力产品化和模块化。

使用低代码平台时，仍然要知道：

- 数据从哪里来；
- 节点之间怎么传递；
- 工具什么时候会失败；
- 知识库如何召回；
- 模型为什么可能编造；
- 复杂项目什么时候需要工程师写代码。

如果项目进入企业级落地，还要考虑内部系统接入、权限体系、数据安全、监控、成本、私有部署和平台依赖。

## 4. 5.2 Coze：快速创建和发布 Agent

原文第一个平台是 Coze。教材里的 Coze 更像低代码/零代码 Agent 搭建平台，通过插件、工作流、知识库、提示词和发布渠道来快速创建智能体。

Coze 的核心定位：

> 像搭积木一样快速创建可交互、可发布的 Agent 或 AI 应用。

它适合：

- AI 应用入门用户；
- 产品经理、运营、个人创作者；
- 想快速验证 Agent 想法的人；
- 需要发布到飞书、微信、豆包等渠道的轻量应用。

### 4.1 Coze 功能模块的版本无关理解

教材截图里的 UI 已经过时，但底层概念仍然有价值。

无论入口怎么变，低代码 Agent 平台通常绕不开这些能力：

- `项目 / Agent`: 最终要创建的智能体或 AI 应用。
- `资源 / 能力组件`: 提示词、知识库、插件、工作流、卡片、数据库等。
- `工作流`: 把任务拆成稳定步骤，例如抓取、筛选、总结、输出。
- `插件 / API / 外部工具`: 连接 RSS、GitHub、arXiv、飞书、数据库等外部能力。
- `知识库`: 让 Agent 基于自己的资料回答。
- `调试 / 评测 / 日志`: 观察每一步是否成功。
- `发布 / 集成`: 把 Agent 从平台 demo 变成用户能用的产品。

### 我的 Coze 卡点

我打开新版 Coze 后发现，它不再像教材截图那样清楚展示“搭积木”的节点入口，而是更像对话式 AI 应用开发。

我的直觉是：

> Coze 现在更像让 AI 帮我搭 AI 应用，表面上更简单，但我更看不见中间到底发生了什么。

这个卡点非常重要，因为它让我意识到：

> 低代码平台不一定天然更透明。越自动化，越需要可观测性来抵消黑盒感。

### 4.2 智能体和应用的区别

`智能体` 更像一个有角色、有能力、有任务边界的 AI 助手。

例如：调研分析师、每日 AI 简报助手、理财参谋、客服助手。

`应用` 更像完整的产品外壳或交互入口。它可以包含一个或多个智能体，也可以包含按钮、表单、页面、数据展示、权限和发布渠道。

关键理解：

> 智能体是能力主体，应用是产品形态。

### 4.3 Coze 的资源库和能力组件

教材说模型决定智能体的下限，资源库决定能力上限。

我的理解：

- 模型像基础大脑，决定理解、推理、总结、写作等基础能力。
- 知识库决定它能基于哪些资料回答。
- 插件/API 决定它能调用哪些外部工具。
- 工作流决定它能否稳定执行多步骤任务。
- 提示词决定角色、边界、格式和行为规则。
- 卡片/界面组件决定输出是否产品化。
- 发布管理决定它能否被真实用户使用。

所以 Agent 不是“模型 + prompt”，而是：

> 模型 + 知识 + 工具 + 工作流 + 提示词 + 存储 + 发布 + 评测组成的产品系统。

### 4.4 “每日 AI 简报”案例

Coze 案例是构建“每日 AI 简报”助手。

它的产品链路：

```text
RSS / GitHub / arXiv 信息源
  -> 插件抓取数据
  -> 参数决定抓什么
  -> 工作流连接数据
  -> 大模型筛选、分类、总结
  -> 结构化 AI 简报
  -> 发布到真实渠道
```

这个案例训练三类能力：

- 多源信息聚合：从媒体、GitHub、arXiv 获取信息。
- 智能体行为定义：用角色设定和提示词控制筛选标准、输出风格和格式。
- 自动化工作流构建：把获取数据、处理内容、格式化输出串起来。

关键理解：

> Agent 要可靠，第一步不是让模型更会写，而是让它拿到正确、及时、相关的数据。

### 4.5 Prompt 在 Agent 中的作用

这一节我问过一个关键问题：现在模型已经很强了，prompt 还重要吗？

最终理解：

> Prompt 不是魔法，但它是 Agent 的行为说明书和输出合同。

在产品级 Agent 中，prompt 仍然重要，因为产品需要稳定、可复用、可解析、可评测的输出。

`System Prompt` 适合放长期稳定规则：

- 你是谁；
- 只关注什么主题；
- 不能输出什么；
- 输出格式必须怎样。

`User Prompt` 在工作流里可以是带变量的任务模板：

```text
请根据 {{articles}}、{{arxiv}}、{{GitHub}} 里的数据，
分别整理成 AI 技术新闻、AI 学术论文、AI 开源项目。
```

这里的变量来自前面插件节点的输出，不是用户随口输入的一句话。

### 4.6 测试、调试与发布

测试“每日 AI 简报”时，不能只看有没有生成文字，还要看：

- 内容准确性；
- 格式完整性；
- 语言风格；
- 数量要求；
- 工具调用是否成功；
- 是否有来源链接；
- 是否出现幻觉。

排查路径：

```text
插件原始数据错 -> 查插件参数
变量传递错 -> 查工作流连接/变量映射
数据是对的但筛选错 -> 改 User Prompt 的筛选标准
格式乱 -> 改 System Prompt 的输出规范
模型编造 -> 加强引用约束和来源校验
```

发布配置也不是装饰。名称、头像、欢迎语、预设问题，都是产品入口设计。

> 后台流程跑得通，不代表用户知道怎么用。欢迎语和预设问题承担 onboarding 和使用引导。

### 4.7 Coze 的优势与局限

优势：

- 低门槛，适合快速上手。
- 插件生态丰富。
- 发布渠道方便，适合飞书、微信、豆包等真实渠道验证。
- 适合原型验证和轻量应用。

局限：

- 灵活性受限。
- 黑盒程度较高，尤其新版对话式入口更明显。
- 平台依赖强。
- 企业级落地时要额外评估安全、权限、监控、成本、API、私有部署和平台变化风险。

我的最终判断：

> Coze 很适合原型验证和用户触达，但不一定适合所有需要强可控、强合规、强工程化的企业级项目。

## 5. 5.3 Dify：可控的 LLM 应用开发与运营底座

Dify 是开源的大语言模型应用开发平台，融合 BaaS 和 LLMOps 理念。

它不是只做一个聊天机器人，而是把原型设计、模型接入、工作流、RAG、工具、插件、部署、日志和运营放在一个平台中。

关键特点：

- `开源`: 可以看代码、自己部署、二次开发。
- `模型中立`: 可以接 GPT、DeepSeek、Llama，或兼容 OpenAI API 的模型。
- `部署灵活`: 可以云端，也可以本地或私有化部署。
- `插件生态`: Marketplace 支持模型、工具、Agent 策略、扩展和捆绑包。
- `企业友好`: 更关注安全、权限、审计、可扩展性。

### 5.1 Dify 和 Coze 的差异

我的最终理解：

> Coze 更像低门槛、对话式、快速发布的 Agent 应用平台；Dify 更像可控、可编排、可部署的 LLM 应用开发平台。

Coze 更适合：

- 快速试想法；
- 低技术门槛使用；
- 快速发布到真实渠道；
- 轻量 Agent 原型。

Dify 更适合：

- 企业级 LLM 应用；
- 需要私有部署；
- 需要接多模型、多工具、多工作流；
- 需要日志、权限、安全和运营能力。

### 5.2 Dify Marketplace

Dify Marketplace 不只是普通插件商店，而是能力组件市场：

- `Models`: 模型接入。
- `Tools`: 外部工具。
- `Agent Strategies`: 智能体执行策略。
- `Extensions`: 平台扩展能力。
- `Bundles`: 一组能力的打包安装。

我曾问：Dify Marketplace 和 Coze 商店有什么区别？

最终理解：

> Dify Marketplace 更像 npm / 插件市场，重点是给开发者装能力组件；Coze 商店更像 App Store / Bot Store，重点是让用户发现和体验成品 Agent。

### 5.3 为什么很多人仍然用 Codex / Claude Code

学习中我问过：既然 Dify、Coze 这些平台插件已经很丰富，为什么很多人还热衷于用 Codex / Claude Code？

最终理解：

> 插件丰富不等于工程问题已经解决。

Coze / Dify / n8n 更像应用搭建平台，适合把已有能力快速拼起来。

Codex / Claude Code 更像工程协作型 coding agent，可以进入代码仓库、文件系统、终端和测试环境，帮我读代码、改代码、跑测试、接 API、写文档、提交 Git。

所以：

- 低代码平台适合快速验证流程。
- Coding agent 适合把关键能力工程化、可控化、可维护化。

### 5.4 Dify 超级助手案例

Dify 案例是构建一个超级智能体个人助手，覆盖：

- 日常生活问答；
- 文案润色优化；
- 多模态内容生成；
- 数据查询与可视化分析；
- MCP 工具集成。

它的架构不是“一个超长 prompt”，而是：

```text
统一入口
  -> 问题分类器
  -> 日常问答 / 文案优化 / 多模态 / 数据分析 / MCP 工具
  -> 对应模块处理
  -> 返回结果
```

关键理解：

> “超级助手”不是让一个模型硬扛所有任务，而是用路由和模块化，把不同类型的问题交给最合适的处理链路。

### 5.5 MCP 在 Dify 中的意义

MCP 可以理解为让模型/Agent 以统一方式连接外部工具和数据源的协议。

我形成的理解：

> MCP 把外部工具包装成标准化服务，让 Agent 平台不用为每个工具单独开发一套适配方式。

但底层仍然会调用工具或服务，MCP 不是让 API 消失，而是让工具接入方式更统一。

### 5.6 Dify 的优势与局限

优势：

- 全栈式开发体验：RAG、工作流、模型管理、工具集成放在一起。
- 低代码与扩展性平衡。
- 企业级安全与合规能力更强。
- 工具集成丰富。
- 开源社区活跃，适合学习和二次开发。

局限：

- 对完全非技术用户有学习曲线。
- 高并发场景可能需要性能优化。
- 多模态不是最强项。
- 企业版成本可能较高。
- API 兼容性可能限制部分第三方集成。

最终理解：

> Dify 更像可控的 LLM 应用开发与运营底座，不是最轻的玩具，但更适合从原型走向企业级应用。

## 6. 5.4 FastGPT：RAG 知识库问答优先

FastGPT 是开源的知识库问答平台与 Agent 构建工具，核心定位是企业级 AI 生产力引擎。

和前面两个平台相比：

- `Coze`: 快速创建和发布。
- `Dify`: 全栈 LLM 应用开发与运营。
- `FastGPT`: 知识库问答和 RAG 优先。

FastGPT 的核心链路：

```text
数据导入 -> 智能分块 -> 向量检索 -> 对话生成
```

我的理解：

> 很多企业场景不需要一开始做复杂多 Agent，只需要把企业知识可靠地接给 LLM。FastGPT 就是围绕知识库问答，把 RAG 链路做得更顺、更可调、更可观测。

### 6.1 为什么 RAG 是企业高频场景

企业有很多稳定资料：

- 公司制度；
- 产品文档；
- 客服知识库；
- 招标文件；
- 内部培训资料；
- 研报、教材、合同。

这些场景的核心不是让 Agent 自己规划复杂行动，而是让 LLM 查到正确资料并基于资料回答。

所以 FastGPT 对企业有吸引力，因为它把文档上传、分块、索引、检索、引用、问答做成核心能力。

### 6.2 RAG 调试：分块不是小事

我学到的重要点：

> 不是把 PDF 上传就万事大吉。RAG 效果很大程度取决于文档怎么被处理。

FastGPT 可以查看分块后的具体内容，这对调试很重要。

分块太长：

- 语义不聚焦；
- 一个片段里混多个主题；
- 检索回来后模型容易抓错重点；
- 成本和上下文压力增加。

分块太短：

- 上下文丢失；
- 单个片段缺少背景；
- 模型拿到后无法完整回答。

标题加入索引、补充索引、图片自动索引等能力，本质是在把文档整理成更适合模型检索和理解的形式。

### 6.3 FastGPT + MCP

FastGPT 虽然主打知识库，但也支持 MCP 工具。

典型组合：

```text
知识库：回答稳定知识、制度、理论、文档内容
MCP 工具：获取实时数据、执行外部服务
工作流：把知识库检索、工具调用和模型生成串起来
```

### 6.4 智能投顾助手案例

原文用 FastGPT 构建“智能投顾助手”。

案例能力：

- 回答金融投资理论问题；
- 查询实时股票行情；
- 做用户风险画像评估；
- 生成投资策略分析报告。

对应能力：

- 知识库：金融理论、概念、案例等稳定资料。
- MCP：实时股票行情、金融数据、图表。
- 工作流：意图识别、分支处理、报告生成。
- 多轮对话：收集风险问卷信息。

这一节我不再记具体投顾角色名字，重点是：

> 行业 Agent 不是一个大 prompt，而是先识别意图，再把不同任务分发给知识库、工具、表单或通用模块。

金融类 Agent 还必须有合规边界：

- 不承诺收益；
- 不提供确定性买卖建议；
- 不编造数据；
- 必须提示市场风险；
- 内容仅供参考；
- 必要时建议咨询持牌专业人士。

### 6.5 FastGPT 的优势与局限

优势：

- RAG 能力强。
- 知识库可调试。
- 文档导入、分块、索引、检索链路完整。
- 支持 MCP 和工作流。
- 适合企业知识助手、客服、制度问答、文档问答。

局限：

- 通用性不如 Dify。
- 发布和分发生态不如 Coze。
- 复杂工程仍需要开发能力。
- RAG 本身仍有难点，例如文档质量、分块策略、召回效果、权限隔离、引用可信度。

最终理解：

> FastGPT 是知识库问答优先的平台。如果任务核心是“让企业文档可问答”，它比通用 Agent 平台更贴近需求。

## 7. 5.5 n8n：业务流程自动化优先

n8n 和前面几个平台不同。它本质上是开源工作流自动化工具，而不是专门的 LLM / Agent 平台。

它的核心不是搭聊天机器人，而是：

> 把 SaaS、数据库、API、消息系统、AI 模型节点连接成自动化业务流程。

典型流程：

```text
业务事件触发
  -> 拉取数据
  -> 条件判断
  -> 必要时调用 LLM
  -> 写入系统
  -> 通知相关人
```

### 7.1 n8n 的基本思维

n8n 的核心是节点和工作流。

- `Trigger 节点`: 流程什么时候开始，例如收到邮件、定时任务、Webhook。
- `Action 节点`: 执行动作，例如发邮件、写数据库、通知飞书。
- `AI / LLM 节点`: 做分类、总结、抽取、生成。
- `Condition 节点`: 按条件走不同分支。
- `Data Transform 节点`: 整理字段、转换格式、合并数据。

关键理解：

> n8n 的主语是业务流程，AI 只是流程中的一个能力节点。

### 7.2 n8n 和 Dify 的差异

我问过：Dify 不是也能做工作流和自动化吗？

最终理解：

> Dify 的主语是 AI 应用，n8n 的主语是业务流程。

Dify 从“用户和 AI 应用交互”出发，再接工具。

n8n 从“业务事件触发流程”出发，中间可以调用 AI。

例如：

- 用户打开 AI 助手问公司制度：优先 FastGPT / Dify。
- 每天自动汇总 CRM 新线索、判断优先级、通知销售：优先 n8n。

### 7.3 智能邮件助手案例

原文用 n8n 搭智能邮件助手。我们没有深入邮件 SOP，而是抽象出平台模式：

```text
收到邮件
  -> 提取发件人、主题、正文
  -> LLM 判断类型和优先级
  -> 按类型分支处理
  -> 生成摘要 / 草稿 / 工单 / 通知
  -> 写入系统或发送消息
```

重要产品判断：

> 不建议一开始让 AI 自动发送所有邮件回复。

原因不是主要因为成本，而是误发风险：

- AI 可能误解邮件意图；
- 可能生成不合适或不合规的内容；
- 客诉、合同、付款、法律、隐私相关邮件需要人工确认；
- 邮件一旦发出就是正式对外沟通。

更安全的策略：

```text
先生成回复草稿 -> 人工审核确认 -> 再发送
```

### 7.4 n8n 的优势与局限

优势：

- 触发器、条件分支、跨系统连接强。
- 适合自动化邮件、CRM、表格、数据库、通知、报表等流程。
- AI 可以插入流程中做理解、抽取、总结、生成。
- 很适合把 AI 接入真实业务系统。

局限：

- 不是最强的 RAG 平台。
- 不是最适合做聊天型 Agent 的平台。
- 复杂流程会变得难维护。
- 仍然需要懂 API、数据结构、权限和错误处理。
- AI 能力依赖外部模型和节点配置。

最终理解：

> 如果任务核心是“让 AI 会回答”，看 Coze / Dify / FastGPT；如果任务核心是“让业务流程自动流转”，看 n8n。

## 8. 四个平台的最终对比

| 平台 | 核心定位 | 最适合的场景 | 主要优势 | 主要风险 |
| --- | --- | --- | --- | --- |
| Coze | 低门槛 Agent 创建与发布 | 原型验证、轻量应用、多渠道触达 | 上手快、发布方便、适合非技术用户 | 黑盒感强、灵活性和企业可控性有限 |
| Dify | 可控的 LLM 应用开发与运营底座 | 企业级 LLM 应用、工作流、RAG、MCP、私有部署 | 开源、模型中立、可部署、可扩展 | 学习曲线高、复杂场景仍需工程能力 |
| FastGPT | RAG 知识库问答优先 | 公司制度问答、客服知识库、文档问答 | 文档处理、分块、索引、检索能力强 | 通用性和发布生态不一定最强 |
| n8n | 业务流程自动化 | 邮件、CRM、表格、数据库、SaaS 自动化 | 跨系统连接强、流程自动执行 | 不是专门 Agent/RAG 平台，复杂流程维护难 |

一句话记忆：

```text
Coze: 快速做出来并发布给用户
Dify: 可控地搭一套 LLM 应用系统
FastGPT: 把企业知识库变成可问答系统
n8n: 把 AI 放进自动化业务流程
```

## 9. 平台选型时我会看的维度

学习到最后，我形成了自己的选型维度：

1. `任务核心是什么`

如果核心是自动化流程，优先 n8n。

如果核心是知识库问答，优先 FastGPT。

如果核心是复杂 LLM 应用和企业级可控，优先 Dify。

如果核心是快速验证和发布给真实用户，优先 Coze。

2. `工具和生态是否匹配`

低代码平台的价值很大一部分来自工具、插件、MCP、知识库和发布渠道。如果平台已经有我需要的地图、搜索、飞书、微信、数据库、MCP 等能力，就能显著降低搭建成本。

3. `发布和用户触达能力`

产品经理需要让用户真实使用，才能验证需求。Coze 这类平台发布到飞书、微信、豆包的能力，对快速验证很有价值。

4. `可控性和黑盒程度`

越自动化、越对话式的平台越容易上手，但也越可能看不见中间过程。需要关注日志、节点、变量、工具调用、错误排查是否可见。

5. `企业级要求`

如果进入企业落地，要看权限、数据安全、隐私合规、审计日志、成本、性能、私有部署、API 稳定性和平台依赖。

## 10. 本章最大的认知变化

这一章我最大的认知变化是：

> 低代码平台不是帮我完全消除技术壁垒，而是在降低上手成本的同时，也可能增加黑盒感和不确定性。

更完整地说：

- 低代码平台可以让我更快验证业务逻辑。
- 但我不能把技术理解完全外包给平台。
- 平台封装越多，我越需要知道它封装了什么。
- 平台越黑盒，我越要关注可观测性和调试路径。
- 原型阶段可以追求快，落地阶段必须关注可控、安全和稳定。

这对 AI 产品经理的启发：

> 产品经理不一定要亲手写所有底层代码，但必须能判断平台边界、设计工作流、定义工具契约、检查输出质量、识别风险，并知道什么时候该从低代码平台转向工程化实现。

## 11. 最终理解

第五章把第四章的底层 Agent 范式放到了真实平台语境里。

第四章让我理解 Agent 内部如何通过模型、工具、观察、反思和循环运行。

第五章让我理解：

> 真实产品里，Agent 不是只靠底层范式，也不是只靠某个大模型，而是要根据业务目标选择合适平台，把模型、知识、工具、流程、发布和评测组织成可用的产品系统。

最终可以用一句话收束：

> 低代码平台的价值不是替代思考，而是加快验证；AI 产品经理的价值不是记住平台按钮，而是知道该用什么平台、为什么用、用到哪里会失控，以及如何把原型推进到可落地的产品。
