## 1. 今天学了什么

### 1.1 Agent 不是“会聊天的模型”

我今天对 Agent 的理解是：Agent 的重点不是知识更多，也不是能不能看图片，而是它有没有围绕目标形成一个行动闭环。

一个更清楚的表达是：

> Chatbot 主要完成一次性问答；Agent 会围绕目标持续感知环境、规划下一步、调用工具执行动作，并根据执行结果调整策略。

Agent 的基本闭环可以理解为：

```text
目标 -> 感知环境 -> 判断当前状态 -> 选择下一步策略 -> 调用工具/执行动作 -> 观察结果 -> 再调整
```

所以，多模态能力不是 Agent 的必要条件。一个只能读文本和 API 的系统也可能是 Agent；一个能看图的模型也未必是 Agent。关键看它是否能基于目标和状态动态行动。

### 1.2 Agent 的四个要素

| 要素 | 我的理解 | A-Level Plus 场景里的例子 |
| --- | --- | --- |
| 环境 Environment | Agent 所处的外部世界 | 学生提交的题目图片、学生对话、题库/知识库、批改规则、历史答题记录、学习报告系统 |
| 传感器 Sensors | Agent 获取信息的入口 | 图片上传、OCR 结果、学生文字输入、历史错题、模型批改结果、规则校验结果 |
| 执行器 Actuators | Agent 改变系统状态或输出结果的手段 | 调用 OCR、调用 LLM 解题、调用 sympy 校验、生成追问、记录错题、生成学习报告、触发人工复核 |
| 自主性 Autonomy | Agent 在边界内判断下一步怎么做 | 根据题型、OCR 置信度、模型一致性决定直接讲解、工具校验、多模型辩论、人工复核或继续追问 |

## 2. 关键纠偏：多模型辩论不等于 Agent

我原来容易把“五个大模型一起辩论”直接称为 Agent，但今天意识到这个说法不够准确。

更准确的理解是：

> 多模型辩论不是完整 Agent，而是 Agent 工作流里的一个校验或仲裁机制。

也就是说：

- 大脑不是“五个模型辩论”本身。
- 大脑是整个任务状态判断和策略选择链路。
- 多模型辩论只是在模型答案冲突、题目风险较高、需要提高可靠性时被触发的一种工具。

可以用这个结构重新表达：

```text
题目输入
-> OCR 和视觉理解
-> 判断题型、OCR 置信度、模型一致性
-> 简单低风险题：直接讲解
-> 计算题/高风险题：调用 sympy 或规则校验
-> 模型结果冲突：触发多模型辩论或人工复核
-> 学生没掌握：生成追问和类似题
-> 写入错题记录和学习报告
```

## 3. A-Level Plus 怎样讲成 Agent

### 3.1 面试版回答

如果面试官问：“你这个到底哪里体现 Agent，而不是普通大模型批改？”

可以这样回答：

> 我最开始确实做的是多模型辩论，但后来意识到辩论只是校验机制，不等于 Agent。真正的 Agent 化设计，是让系统根据题型、OCR 置信度、模型一致性和学生反馈，动态选择下一步策略：简单题直接讲解，高风险计算题调用规则或 sympy 校验，模型冲突时触发辩论或人工复核，学生没掌握时继续追问并生成类似题。

这比“我用了五个模型辩论”更准确，因为它强调的是目标、状态判断、工具调用和反馈闭环。

### 3.2 三个优先判断信号

如果当前阶段只能先做三个判断信号，我会优先选：

| 判断信号 | 为什么重要 | 下一步策略 |
| --- | --- | --- |
| OCR 置信度 | 输入错了，后面讲解都会错 | 低置信度时要求学生重拍、确认题目，或进入人工复核 |
| 题型 | 不同题型需要不同批改方式 | 计算题走规则/sympy 校验，图像题走更强视觉理解，开放题走 rubric 评分 |
| 模型一致性 | 多模型不一致通常意味着风险更高 | 一致时低成本模型生成讲解；不一致时触发辩论、工具校验或人工复核 |

学生历史错题、学生追问内容、是否真的掌握，可以放在下一阶段做，因为这部分会明显增加产品复杂度和响应时间。

## 4. 规则写死了，还算智能吗？

今天也澄清了一个重要问题：Agent 不是完全不需要规则。真实产品里的 Agent 往往更需要规则边界。

如果面试官质疑：“你这个流程都是规则写死的，哪里有智能？”

更好的回答是：

> 在教育批改场景里，我不会追求完全放任模型自由决策，因为答案可靠性比炫技更重要。规则负责定义安全边界和确定性判断，LLM 负责理解题目、解释思路和生成互动，Agent 的智能体现在它能根据当前状态在多个策略之间做选择，比如直接讲解、调用工具校验、触发多模型辩论、要求重拍或升级人工复核。

核心观点：

- 规则不是反智能，而是产品安全边界。
- LLM 不是万能判断器，更适合做理解、解释和互动。
- 对确定性强的数学判断，需要规则引擎或 sympy 兜底。
- Agent 的价值在于“状态判断 + 策略选择 + 工具调用 + 反馈闭环”。

## 5. OCR 失败的兜底设计

OCR 是拍照批改的上游，如果题目识别错了，后续模型讲解可能全错。因此不能只依赖 OCR 输出。

可行的兜底流程：

```text
OCR 输出题目文本和置信度
-> 视觉模型独立复述题目
-> 对比 OCR 文本和视觉模型复述是否一致
-> 如果不一致，让学生确认题目或重新拍照
-> 如果关键数字、符号、图像条件不清晰，禁止直接批改
-> 进入人工复核或重拍流程
```

产品原则：

> 低置信度不继续生成答案，高风险输入先澄清。

## 6. 今天的卡点

| 卡点 | 今天的理解 |
| --- | --- |
| 多模型辩论是不是 Agent | 不是完整 Agent，而是 Agent 工作流中的校验/仲裁机制 |
| 是否必须接入 Claude Code 或 Codex 才算 Agent | 不需要。Claude Code/Codex 是通用软件工程 Agent；A-Level Plus 应该是垂直教育批改 Agent |
| 规则写死是否就不智能 | 不一定。教育产品里规则是安全边界，智能体现在系统能根据状态选择策略并调用工具 |
| 当前 demo 的不足 | 学生是否真正掌握、是否自动出类似题、学习闭环还没有完全做出来，目前更像 demo，需要后续补强 |
