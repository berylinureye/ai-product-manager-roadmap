# Problem Space vs. Solution Space：先定义问题，再设计方案

## 一句话主旨

产品经理不要直接做用户提出的方案，而要先回到问题领域，判断用户真正的痛点是什么，以及这个问题是否值得解决。

## 核心框架

```text
用户表达的需求 / 方案
-> 追问背后的真实问题
-> 判断是否是值得解决的机会
-> 再进入解决方案设计
```

文章的重点不是反对做方案，而是强调方案必须建立在对问题的清晰理解之上。

## 1. 资源有限，所以不能什么都做

文章开头指出，产品团队经常跳过前面的思考，直接开始做用户或团队提出的解决方案。

但现实是资源永远有限。产品经理的责任不是把所有需求都实现，而是判断：

- 哪些问题真实存在；
- 哪些问题足够重要；
- 哪些问题值得公司投入资源解决；
- 哪些方案只是表面需求。

产品思维重点：

> PM 的价值不是把需求搬进排期，而是判断什么问题值得被解决。

## 2. Problem 和 Opportunity 的区别

文章区分了 problem 和 opportunity：

| 概念 | 含义 |
| --- | --- |
| Problem | 目标用户还没有被满足的需要、欲望或痛点 |
| Opportunity | 已验证、同时能让用户和公司受益的商业机会 |

一个问题真实存在，不代表它一定是好的产品机会。它还必须同时满足两侧价值：

- 用户价值：用户真的需要，愿意使用，觉得它解决了痛点。
- 公司价值：解决这个问题能带来商业回报或战略价值。

例子：如果用户只需要付很低价格就能无限听音乐，用户可能很喜欢，但如果公司无法覆盖成本，就不是可持续的机会。

## 3. 用户擅长描述痛点，不一定擅长设计方案

文章提醒，不要只问用户“你想要什么功能”。因为用户往往能描述自己的痛苦、场景和不满，但不一定能设计出最合适的解决方案。

经典例子是：用户可能说想要“更快的马”，但背后的真实问题其实是“想更快到达目的地”。

产品思维重点：

> 用户提出的 solution 是线索，不是结论。PM 要从表面方案反推真实问题。

## 4. Problem Space：问题领域

Problem Space 是由用户定义的。这里关注的是用户的事实、痛点和感受，而不是马上讨论功能。

用户研究时要理解：

- 什么事情造成困扰；
- 用户生活或工作流程实际如何发生；
- 用户在过程中有什么情绪、压力或担心。

在 Problem Space 里，产品经理应尽量避免过早提出方案。目标是还原用户真实处境。

## 5. Solution Space：解题领域

Solution Space 是在理解问题之后，开始讨论如何解决问题。

这里才进入：

- 功能设计；
- 流程设计；
- 技术方案；
- 交互方案；
- 实施步骤。

如果问题还没定义清楚，团队就进入 Solution Space，很容易出现一种情况：大家围绕某个方案争论很久，但其实没有确认这个方案是否解决了真实问题。

产品思维重点：

> Problem Space 决定是否值得做；Solution Space 决定怎么做。

## 6. 案例：单车通勤族

文章用旧金山单车通勤族举例。

背景：

- 成功定义：在获利的同时减少通勤产生的碳足迹。
- 目标用户：在旧金山通勤上班的人。

用户表面上可能说：

```text
如果有电动单车，我会更愿意骑车上下班。
```

直觉上，团队可能以为问题是“骑车太累”，于是开始设计电动单车方案。

但深入理解后，可能发现：

- 用户其实喜欢骑车，因为可以运动；
- 用户也喜欢通勤路上的风景；
- 真正的问题是到办公室后满身汗，担心显得不专业。

这时，解决方案就不一定是电动单车，可能是：

- 办公室淋浴间；
- 换衣设施；
- 通勤后的整理空间；
- 与办公场景配套的服务。

这个案例说明：

> 用户说出的方案，可能只是对自己痛点的第一次猜测。PM 要继续追问，找到真正影响用户行为的原因。

## 7. 内部产品的特殊困境

文章最后谈到内部产品。内部产品的用户往往是同事，访谈和沟通更方便，但也更容易受到组织压力影响。

内部产品团队常被当成 IT 或需求实现团队，所以容易直接照做业务方或老板提出的功能。

但产品经理仍然要追问：

- 这些功能是否解决用户真正需要解决的问题？
- 这些功能是否走在长期产品路线图上？
- 这些功能和现有产品能力是否可以共生？
- 它会不会只是增加系统复杂度？

文章给出的更好流程是：

1. 先做用户访谈，理解潜在痛点。
2. 把痛点带回团队，讨论解决方案。
3. 快速验证方案是否真的解决痛点。

## 常见误区

1. 把用户提出的功能当成真实需求。
2. 还没理解问题，就开始讨论方案。
3. 只判断用户是否喜欢，不判断公司是否能获得价值。
4. 内部产品中，因为老板或业务方强势，就放弃问题探索。
5. 把“做出来”当成交付标准，而不是把“解决问题”当成标准。

## 可复用表达

- 用户提出的功能需求是线索，不是结论。
- Problem Space 里先还原事实、痛点和感受；Solution Space 里再讨论方案和实现。
- 好的产品机会要同时满足用户价值和公司价值。
- PM 不只是接需求的人，而是负责判断需求背后问题是否值得解决的人。
- 如果团队一开始就在争论方案，PM 应该把讨论拉回问题本身。

## 读后收获

这篇文章最重要的产品思维是：先判断问题，再设计方案。

它提醒产品经理，不要被用户或组织给出的表面 solution 带着走，而要先弄清楚：

- 目标用户是谁；
- 他们真实困扰是什么；
- 这个困扰在什么场景下发生；
- 它是否足够重要；
- 解决它是否同时对用户和公司有价值；
- 当前提出的方案是否只是众多可能方案之一。
