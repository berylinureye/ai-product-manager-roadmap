## 2. Agent 的基本定义

Agent 不是简单的“会聊天的模型”，而是一个能够围绕目标与环境交互的系统。

核心闭环：

```text
目标 -> 感知环境 -> 思考下一步 -> 执行动作 -> 观察结果 -> 再调整
```

Agent 的四个基础要素：

| 要素 | 含义 | 例子 |
| --- | --- | --- |
| Environment | Agent 所处的环境 | GitHub 项目页、README、issue、用户学习目标 |
| Sensors | Agent 获取信息的入口 | 用户输入链接、网页读取器、GitHub API、搜索工具 |
| Actuators | Agent 能执行的动作 | 打开网页、调用 API、写入 Markdown、生成推荐 |
| Autonomy | Agent 根据目标和状态自主决定下一步 | 判断是否继续搜索、是否调用工具、是否结束任务 |

我今天纠正的理解：

> 多模态能力不是 Agent 的必要条件。一个能看图的模型不一定是 Agent；一个只读文本和 API 的系统也可能是 Agent。关键看它有没有目标驱动的行动闭环。

## 3. 传统智能体的演进

传统智能体不是一开始就很复杂，而是从简单规则逐步叠加状态、目标、效用和学习能力。

| 类型 | 它在回答的问题 | 我的理解 |
| --- | --- | --- |
| 简单反射智能体 | 当前发生了什么，我按规则做什么？ | 只看当前输入，不记过去，不想未来 |
| 基于模型的反射智能体 | 世界现在是什么状态？ | 多了内部状态，可以记住或估计环境状态 |
| 基于目标的智能体 | 我该做什么才能达成目标？ | 能围绕目标规划行动路径 |
| 基于效用的智能体 | 哪个方案最值得？ | 能在时间、成本、风险、体验之间权衡 |
| 学习型智能体 | 下次怎样做得更好？ | 能根据反馈优化策略 |

关键理解：

> Agent 的演进不是突然从规则跳到大模型，而是从规则反应逐步走向状态理解、目标规划、效用权衡和经验学习。

## 4. LLM Agent 的新范式

传统智能体的能力更多依赖工程师显式设计规则、状态模型、目标函数或效用函数。LLM Agent 的变化是：它可以用自然语言理解模糊目标，并临时拆解任务、选择工具、根据上下文调整路径。

以智能旅行助手为例：

```text
用户：帮我规划一次厦门三天旅行，预算中等，不要太累，适合拍照。

LLM Agent:
-> 理解自然语言目标
-> 拆成确认偏好、查天气、查交通、找景点、控预算
-> 调用天气/地图/搜索/酒店等工具
-> 根据用户反馈动态修正方案
```

LLM Agent 的三类核心能力：

| 能力 | 含义 |
| --- | --- |
| 规划与推理 | 把高层目标拆成可执行子任务 |
| 工具使用 | 识别信息缺口，并调用外部工具补齐真实信息 |
| 动态修正 | 把用户反馈或环境变化变成新约束，调整后续行动 |

重要理解：

> LLM Agent 更灵活，但稳定性和可控性更弱，因此更需要工具、规则和观察反馈来兜底。

## 5. 智能体的三种分类方式

### 5.1 按内部决策架构分类

这一类对应传统智能体演进：

```text
简单反射 -> 基于模型 -> 基于目标 -> 基于效用 -> 学习型
```

### 5.2 按时间和反应性分类

| 类型 | 特点 | 产品取舍 |
| --- | --- | --- |
| 反应式 Agent | 收到刺激后快速响应 | 快、便宜、稳定，但短视 |
| 规划式 Agent | 行动前先做复杂规划 | 能处理复杂任务，但慢、贵、容易中途出错 |
| 混合式 Agent | 简单任务快反应，复杂任务慢规划 | 更接近真实产品，但设计更复杂 |

我今天的理解：

> 真实产品通常需要混合式 Agent。简单问题直接回答，复杂问题再规划；否则所有任务都深度规划，会导致慢、贵、体验差。

### 5.3 按知识表示分类

| 类型 | 核心 | 优点 | 风险 |
| --- | --- | --- | --- |
| 符号主义 AI | 明确规则和逻辑推理 | 可解释、可追溯、适合确定性判断 | 规则写不完，遇到模糊场景脆弱 |
| 亚符号主义 AI | 神经网络从大量数据中学习模式 | 适合图片、语音、自然语言、语义理解 | 黑箱、可能幻觉、逻辑不稳定 |
| 神经符号主义 AI | 神经网络 + 规则/工具/结构化步骤 | 兼顾理解能力和可靠性 | 系统设计更复杂 |

今天纠正的误区：

> README 总结属于亚符号主义能力，因为它需要理解自然语言和项目语义。README 中的固定字段检查、链接校验、stars 阈值判断，则更适合符号主义规则。

用于 AI 产品周榜分析 Agent：

- 亚符号主义适合：总结 README、提炼用户场景、判断产品价值、把技术内容翻译成产品经理语言。
- 符号主义规则适合：校验 stars、更新时间、链接可访问性、输出结构完整性、固定评分规则。

## 6. PEAS：任务环境定义

PEAS 是定义 Agent 任务环境的框架：

```text
P = Performance 性能度量：怎样算做得好
E = Environment 环境：Agent 工作的世界
A = Actuators 执行器：Agent 能做什么动作
S = Sensors 传感器：Agent 通过什么入口获取信息
```

产品经理视角：

> PEAS 不是技术细节表，而是 Agent 产品定义表。它能逼我提前想清楚成功指标、工作场景、动作边界、信息来源和兜底点。

AI 产品周榜分析 Agent 的 PEAS：

| PEAS | 内容 |
| --- | --- |
| P 性能度量 | 推荐是否准确、是否能形成学习笔记、用户是否采纳、耗时是否可接受 |
| E 环境 | GitHub 周榜、项目页、README、issue、docs、用户学习目标 |
| A 执行器 | 打开网页、读取 README、调用 GitHub API、比较项目、生成笔记 |
| S 传感器 | 用户输入链接、网页读取结果、API 返回字段、搜索结果、用户反馈 |

## 7. 任务环境的复杂性

Datawhale 提醒：LLM Agent 通常不是在完美环境里工作，而是在复杂数字环境里工作。

| 特性 | 含义 | GitHub 周榜分析例子 |
| --- | --- | --- |
| 部分可观察 | Agent 看不到全部事实 | README 不等于真实使用体验，stars 不等于真实价值 |
| 不确定性 | 同一个动作结果可能不稳定或不可靠 | API 可能失败，搜索排序可能变，LLM 两次总结重点不同 |
| 多行动者 | 环境里还有其他人或系统在改变状态 | 维护者更新代码，用户提交 issue，榜单算法变化 |
| 序贯且动态 | 前一步影响后一步，环境还会变化 | 项目分类错了，后续推荐和笔记都会错 |

关键区分：

```text
部分可观察 = 信息不完整。
不确定性 = 信息或行动结果不稳定、不一定可靠。
动态环境 = 世界本身正在变化。
```

产品原则：

> 不要让 Agent 把不确定的判断包装成确定结论。需要置信度、交叉验证、错误提示和补充搜索。

## 8. Agent Loop：智能体运行机制

Agent 不是一次性完成任务，而是通过循环逐步推进。

```text
Perception 感知
-> Thought 思考
-> Action 行动
-> Environment State Change 环境状态变化
-> Observation 新观察
-> 下一轮 Perception
```

几个概念的区别：

| 概念 | 含义 |
| --- | --- |
| Observation | 环境或工具返回的结果反馈 |
| Perception | Agent 接收并理解反馈的过程 |
| Thought | Agent 基于当前状态决定下一步 |
| Action | Agent 执行的具体动作 |

例子：

```text
Action：调用 GitHub API 查询项目更新时间
Observation：API 返回 pushed_at = 2026-05-20
Perception：Agent 理解为“项目最近仍活跃”
Thought：这个项目可以进入候选分析
```

关键理解：

> Action 不是终点。Action 执行后必须检查 Observation，否则 Agent 不知道动作是否真的成功。

## 9. TAO 交互协议：Thought-Action-Observation

这一节解释了为什么 Agent 不是光靠 prompt，而是需要外部程序把 LLM 的输出解析并执行。

LLM 不能只说：

```text
我需要查一下北京天气。
```

因为这只是自然语言，程序不知道调用哪个工具、传什么参数。它需要输出可解析格式：

```text
Thought: 用户想知道北京天气，我需要调用天气工具。
Action: get_weather(city="北京")
```

外部 Parser 会解析 Action：

```text
tool_name = get_weather
args = { city: "北京" }
```

工具执行后，结果被包装成 Observation：

```text
Observation: 北京当前天气为晴，气温 25 摄氏度，微风。
```

完整链路：

```text
LLM 输出 Thought + Action
-> Parser 解析 Action
-> Tool Executor 执行工具
-> Environment 返回结果
-> Perception 包装成 Observation
-> Observation 放回上下文
-> LLM 继续 Thought + Action
```

常见风险和兜底：

| 风险 | 兜底 |
| --- | --- |
| 模型没按格式输出 | 输出格式校验，返回错误 Observation |
| 模型调用不存在的工具 | 工具白名单校验，不执行，要求重新选择 |
| 工具参数错误 | 参数校验，提示可用参数 |
| JSON 过长或太复杂 | 包装成简洁 Observation |
| 无限循环 | 最大循环次数和 Finish 结束条件 |

## 10. 5 分钟智能旅行助手案例

第一章最后用一个旅行助手说明最小 Agent 怎么跑起来。

用户任务：

```text
请帮我查询今天北京的天气，然后根据天气推荐一个合适的旅游景点。
```

这个任务不能直接一次性回答，因为景点推荐依赖天气结果：

```text
晴天 -> 推荐户外景点
暴雨 -> 推荐室内景点
```

最小 Agent 的四个组件：

| 组件 | 作用 |
| --- | --- |
| System Prompt | 定义角色、工具、输出格式和结束规则 |
| Tools | 提供真实能力，如 get_weather、get_attraction |
| LLM Client | 调用模型，让模型生成下一步 Thought/Action |
| Main Loop | 保存历史、解析 Action、执行工具、写回 Observation、判断是否 Finish |

运行过程：

```text
第 1 轮：
Thought: 需要先获取北京天气。
Action: get_weather(city="北京")
Observation: 北京晴，26 摄氏度。

第 2 轮：
Thought: 已知道北京晴天，适合户外景点。
Action: get_attraction(city="北京", weather="Sunny")
Observation: 推荐颐和园或长城。

第 3 轮：
Thought: 已经有天气和景点信息，可以回答用户。
Action: Finish[今天北京晴，适合去颐和园...]
```

关键理解：

- `prompt_history` 很重要，因为下一轮模型需要知道前面查到了什么。
- `Finish[最终答案]` 是停止信号，否则 Agent 可能一直循环调用工具。
- 工具调用不是为了“看起来像 Agent”，而是因为任务本身有依赖关系和真实信息缺口。

## 11. 今天容易混淆但已经纠正的点

| 混淆点 | 纠正后的理解 |
| --- | --- |
| 会总结网页就是 Agent | 不一定。单步总结只是 AI 功能，Agent 需要目标、行动、观察和下一步决策 |
| 多模态模型就是 Agent | 不一定。多模态只是传感器能力，关键看有没有行动闭环 |
| README 总结不是亚符号主义 | 错。README 总结需要语义理解，属于亚符号主义能力 |
| Observation 等于用户反馈 | 不完全。用户反馈可以是 Observation，但工具返回、网页 404、API 结果也都是 Observation |
| Parser 是思考模块 | 错。Parser 是外部程序，用来解析 Action 并交给工具执行 |
| Finish 只是输出格式 | 不只是格式，它是 Agent 停止循环的信号 |

## 12. 可复用表达

### 12.1 Agent 和 Chatbot 的区别

> Chatbot 主要完成一次性问答；Agent 会围绕目标持续感知环境、规划下一步、调用工具执行动作，并根据执行结果调整策略。

### 12.2 LLM Agent 为什么需要工具

> LLM 擅长理解目标和规划步骤，但它不是实时世界本身。天气、价格、项目状态、网页内容等需要通过工具获取；数学计算、格式校验、状态判断也需要规则和工具兜底。

### 12.3 神经符号混合

> LLM 负责理解语义和生成内容，规则/API/工具负责确定性校验和真实信息获取。成熟 Agent 往往不是纯神经网络，而是神经网络、结构化工具和符号规则的组合。

### 12.4 Agent 产品设计原则

> Agent 产品设计不是让模型自由发挥，而是定义清楚目标、环境、可用工具、成功指标、观察反馈、错误兜底和停止条件。
